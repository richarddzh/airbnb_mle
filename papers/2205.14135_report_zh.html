论文解读：FlashAttention — Fast and Memory-Efficient Exact Attention with IO-Awareness

论文信息
作者：Tri Dao, Daniel Y. Fu, Stefano Ermon, Atri Rudra, Christopher Ré
机构：Stanford University, University at Buffalo (SUNY)
发表时间：2022 年 6 月（NeurIPS 2022）
arXiv：2205.14135，PDF：2205.14135.pdf
主题：Transformer 加速、注意力机制、GPU IO 优化、CUDA kernel、长序列建模

一句话概括

FlashAttention 把标准 attention 重写成一个"按块计算 + 不落盘中间矩阵"的融合 CUDA kernel，结果完全等价（不是近似！），但训练 Transformer 速度快 2–4 倍，显存占用从 O(N²) 降到 O(N)。它的核心思想不是减少 FLOPs，而是减少 GPU 显存（HBM）的读写次数 —— 即"IO-Aware"算法设计。

为什么初学者要读这篇论文

  • 它是现代 LLM 训练/推理的事实标准：PyTorch 2.0 的 F.scaled_dot_product_attention、HuggingFace、vLLM、几乎所有大模型框架背后都在用 FlashAttention（及其后续版本 v2/v3）。
  • 它教你一个非常重要的工程思想：在 GPU 上跑得快不等于 FLOPs 少，而是 memory access 少。这条原则在很多场景下都适用。
  • 它的算法本身（online softmax + tiling）很优雅，理解之后你看 LLM 加速领域很多论文都会容易很多。

背景知识：你需要先理解什么

1. 标准 Attention 是怎么算的

给定 Query、Key、Value 三个矩阵 Q、K、V ∈ ℝN×d（N 是序列长度，d 是每个 head 的维度），标准的注意力计算分三步：

S = Q K⊤ ∈ ℝN×N
P = softmax(S) ∈ ℝN×N
O = P V ∈ ℝN×d

问题在于：当 N 很大（比如 4096、16K、64K）时，S 和 P 这两个 N×N 的矩阵巨大无比。比如 N=16K，一个 fp16 的 P 矩阵就是 16K × 16K × 2 字节 = 512 MB —— 这还只是一个 head 一个 batch！

2. GPU 显存的两个层级

这是理解 FlashAttention 的关键。GPU 上有两种主要的"内存"：

层级容量（A100 为例）带宽类比
HBM（High Bandwidth Memory，显存）40–80 GB~1.5 TB/s大但慢，相当于"硬盘"
SRAM（on-chip，每个 SM 的共享内存/寄存器）≈ 192 KB / SM~19 TB/s（10 倍以上）小但快，相当于"CPU 缓存"

💡 平时我们说"显存不够"指的是 HBM。但程序跑得快不快，往往取决于 从 HBM 搬数据到 SRAM 搬了多少次。这就是"memory-bound"（受内存带宽限制）的含义。在现代 GPU 上，计算速度远远快于 HBM 带宽，所以很多 op 的瓶颈是 IO 而不是算力。

3. 标准实现的 IO 浪费

PyTorch 的标准 attention 实现会：

  1. 从 HBM 读取 Q、K → 算出 S，把 S 写回 HBM（N² 大小）。
  2. 从 HBM 读 S → 算 softmax 得到 P，把 P 写回 HBM（又一次 N²）。
  3. 从 HBM 读 P 和 V → 算 O，写回 HBM。

每一步都要把巨大的 N×N 矩阵在 HBM 上读一遍、写一遍。HBM 访问总量是 O(N² + Nd)，绝大部分时间都在等内存。

核心思想：能不能不把 N×N 矩阵写到 HBM？

FlashAttention 的目标：S 和 P 这两个大矩阵，根本不要在 HBM 里出现，全程只在 SRAM 里以小块的形式存在。

这听起来很难，因为：

  • Softmax 需要看到整行。因为分母 ∑ⱼ exp(Sᵢⱼ) 是对一整行求和。如果按块处理，怎么算分母？
  • Backward 需要 S 和 P 来算梯度。如果不存它们，反向怎么办？

FlashAttention 用两个技巧分别解决：Tiling（分块计算 + online softmax） 和 Recomputation（反向重算）。

技巧 1：Tiling 与 Online Softmax

从"数值稳定的 softmax"开始

为了避免 exp 溢出，softmax 通常这样算（叫 safe softmax）：

m(x) = maxi xi
f(x)i = exp(xi − m(x))
ℓ(x) = ∑i f(x)i
softmax(x)i = f(x)i / ℓ(x)

关键观察：softmax 可以"边算边合并"

假设把向量 x 拆成两段 x(1)、x(2)。如果我们对每段先算出局部的 (m, ℓ)，能不能合并成整段的 (m, ℓ) ？答案是可以：

m = max(m(1), m(2))
ℓ = em(1)−m · ℓ(1) + em(2)−m · ℓ(2)

意思是：发现一个新的更大的最大值 m 时，把之前累计的 ℓ 用一个修正因子 em_old − m_new rescale 一下，再加上新块的贡献。这就是 online softmax（最早来自 Milakov & Gimelshein 2018）。

把这个思想推广到 Attention 的输出 O

不仅 softmax 的归一化因子可以"边算边合并"，最终的输出 O = P·V 也可以。FlashAttention 在 SRAM 里维护三个东西：

  • Oi：当前累计的输出（每行 d 维）
  • mi：当前行已见过的最大值
  • ℓi：当前行已累计的归一化分母

每当处理新的一块 Kj, Vj 时，就更新这三个量。

算法 1（简化版伪代码）

输入：Q, K, V 在 HBM 中，SRAM 大小 M
块大小：B_c = M/(4d),  B_r = min(M/(4d), d)
把 Q 切成 T_r 块，把 K、V 各切成 T_c 块
在 HBM 初始化 O=0, ℓ=0, m=-∞

for j = 1..T_c:                       # 外层：遍历 K, V 的块
    把 K_j, V_j 从 HBM 加载到 SRAM
    for i = 1..T_r:                   # 内层：遍历 Q 的块
        把 Q_i, O_i, ℓ_i, m_i 加载到 SRAM
        S_ij = Q_i · K_j^T            # 一小块 attention score (B_r × B_c)
        m̃_ij = rowmax(S_ij)
        P̃_ij = exp(S_ij − m̃_ij)
        ℓ̃_ij = rowsum(P̃_ij)

        # 用 online softmax 公式合并新旧统计量
        m_new = max(m_i, m̃_ij)
        ℓ_new = e^(m_i − m_new) · ℓ_i + e^(m̃_ij − m_new) · ℓ̃_ij

        # 更新输出（也要 rescale 旧的 O_i）
        O_i = diag(ℓ_new)^(-1) · (
                diag(ℓ_i) · e^(m_i − m_new) · O_i
                + e^(m̃_ij − m_new) · P̃_ij · V_j
              )
        把 O_i, ℓ_new, m_new 写回 HBM
return O

💡 直觉解读：外层循环遍历 K、V 的块（"对每一段 key 都看一次所有 query"）。每次见到新的 key 块时，更新所有受影响行的局部 max、局部分母 ℓ，并把之前累计的 O 用修正因子缩放后加上新贡献。整个过程 从来没有把 N×N 的 S 或 P 完整写到 HBM，只有 O（N×d）、ℓ（N）、m（N）这些"瘦"的东西在 HBM 来回搬。

技巧 2：Backward 的 Recomputation

反向传播需要 S 和 P 才能算梯度。直接存这两个矩阵会变回 O(N²) 显存，这就违背初衷了。

FlashAttention 的做法：只保存 O 和归一化统计量 (m, ℓ)（都是 O(N) 大小），反向时按需重新计算 S 和 P。由于重算的过程也是 tile-based 的，全程都在 SRAM 里，速度反而比从 HBM 读巨大的 P 还要快。

💡 这其实是"选择性 gradient checkpointing"的思想：用一点点额外的 FLOPs（重算）换大量的 HBM 读写减少。常规 gradient checkpointing 是省显存但变慢；FlashAttention 的 recomputation 是既省显存又变快，因为瓶颈本来就在 IO。

IO 复杂度分析（理论亮点）

实现HBM 访问次数额外显存
标准 attentionΘ(Nd + N²)O(N²)
FlashAttentionΘ(N² d² / M)O(N)

典型 d=64–128，M ≈ 100 KB，所以 d² / M 远小于 1，等于说 FlashAttention 的 HBM 访问比标准实现少了好多倍（论文实测最多 9×）。论文还证明了这是 渐进意义上最优的：在某个 M 范围内，没有任何精确 attention 算法能进一步降低 HBM 访问数量。

Block-Sparse FlashAttention（顺带的扩展）

把 FlashAttention 和 分块稀疏 mask 结合：mask 为 0 的块直接跳过不算。如果稀疏率是 s，HBM 访问大约下降到原来的 s 倍。论文用这个跑通了 64K 序列长度。

实验结果亮点

训练速度

  • BERT-large：比 MLPerf 1.1 训练记录快 15%。
  • GPT-2 small/medium：相比 HuggingFace 实现快 3×，相比 Megatron-LM 快 1.7×。
  • Long-Range Arena：比 vanilla Transformer 快 2.4×。

长序列能力

  • GPT-2 上下文从 1K 扩到 4K，速度仍比 Megatron 1K 上下文快 30%，困惑度（PPL）还更好 0.7 个点。
  • 首次让 Transformer 在 Path-X（序列长 16K）上做到非随机性能（61.4% 准确率），block-sparse 版本进一步攻克 Path-256（64K）。

注意力本身的 benchmark

  • Forward + Backward 比 PyTorch 快最多 3×。
  • 显存占用最多省 20×，且随 N 线性增长（不是 N²）。

实战中怎么用 FlashAttention

对于普通用户，不需要写 CUDA。下列方式都自动用到了 FlashAttention（或它的后续版本）：

import torch.nn.functional as F

# PyTorch 2.0+ 自动选择 FlashAttention 实现
out = F.scaled_dot_product_attention(q, k, v, is_causal=True)
  • PyTorch ≥ 2.0：F.scaled_dot_product_attention 在合适条件下自动用 FlashAttention。
  • HuggingFace Transformers：在加载模型时传 attn_implementation="flash_attention_2"。
  • vLLM、TensorRT-LLM、SGLang 等推理框架默认就是 FlashAttention。
  • 独立库：pip install flash-attn（Tri Dao 维护的官方仓库）。

⚠️ 要求：FlashAttention 对 GPU 架构有要求（一般需要 Ampere / Ada / Hopper，即 A100、RTX 30/40 系、H100），且 dtype 通常是 fp16 或 bf16。fp32 下用不上。

常见疑问 Q&A

Q1：FlashAttention 是近似算法吗？

不是。它是 exact attention，输出和标准实现完全一致（在浮点误差范围内）。它只是改变了计算顺序和内存访问模式。

Q2：FLOPs 没变，为什么会更快？

因为现代 GPU 的瓶颈是内存带宽，不是算力。FLOPs 甚至增加了一点（因为反向重算），但 HBM 读写少了很多倍，所以整体快很多。

Q3：和"线性 attention"（如 Performer、Linformer）什么关系？

线性 attention 是近似方法，把复杂度从 O(N²) 降到 O(N)，但效果会损失，而且在中等长度下经常比标准 attention 还慢（因为常数大）。FlashAttention 是精确方法，FLOPs 仍然是 O(N²)，但常数极低，wall-clock 速度反而比大多数近似方法都快，效果还完全一致。

Q4：FlashAttention v1 / v2 / v3 有什么区别？

这篇是 v1。后续：v2（2023）优化了 work partitioning，比 v1 再快 2×；v3（2024，针对 H100）利用了 Hopper 架构的异步 TMA 和 FP8，再快 1.5–2×。核心思想没变，都是 tiling + online softmax + recomputation。

Q5：算法里 em_old − m_new 这个 rescale 是什么意思？

之前累积 ℓ 和 O 时，是基于"局部最大值 m_old"减去过的。现在出现了一个更大的 m_new，相当于所有之前的 exp 项都要乘上 exp(m_old − m_new) 才能跟新块的指标对齐。这个修正因子保证了无论分多少块、按什么顺序处理，最终结果和一次性算 softmax 完全一致。

FlashAttention 留下的工程启示

  1. "FLOP 少 ≠ 跑得快"。在 GPU 上设计算法，必须考虑 memory hierarchy。
  2. 融合 kernel（kernel fusion）的威力。把"matmul + mask + softmax + dropout + matmul"合成一个 CUDA kernel，避免中间结果落盘，这种思路可以推广到很多其他 op（layer norm + dropout + 残差等）。
  3. Recomputation 不一定慢。当瓶颈是 IO 时，"多算一点、少存一点"反而更快。
  4. 新的硬件抽象层。FlashAttention 之后催生了 Triton、CUTLASS、ThunderKittens 等让用户更容易写 IO-aware kernel 的工具。
