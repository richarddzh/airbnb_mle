解读：Accelerating Large-Scale Test Migration with LLMs

原文：Accelerating Large-Scale Test Migration with LLMs
主题：LLM 代码迁移、Enzyme 到 React Testing Library、自动验证、长上下文。

一句话总结

Airbnb 将 LLM 放进验证驱动的迁移流水线中，用 state machine、retry loops、动态错误反馈和长上下文提示，在 6 周内完成近 3,500 个测试文件迁移，替代原估 1.5 年手工工作。

问题背景

Airbnb 自 2020 起新测试转向 React Testing Library，但遗留大量 Enzyme 测试。不能直接删除，因为会损失覆盖率；也不能简单替换 API，因为 Enzyme 与 RTL 的测试理念不同。

工程方案

设计做法意义
State machine每个文件依次经过 refactor、fix Jest、fix lint、fix tsc、complete。将复杂迁移拆成可验证阶段。
Validation-first每步先跑验证，失败才调用 LLM 修复。用确定性工具约束 LLM 输出。
Retry loops把错误和最新文件内容回填给 LLM，最多多轮尝试。比一次性 prompt 更可靠。
Long context复杂文件 prompt 40k-100k tokens，含组件源码、相关测试、few-shot 示例。让 LLM 学到团队局部模式。
Sample, tune, sweep抽样失败模式，改 prompt/脚本，小样本验证后全量扫。系统性处理长尾。

结果

首次 bulk run 在 4 小时内自动迁移 75%；4 天调优后达到 97%；剩余 3% 人工基于 LLM baseline 修复。最终 6 周完成迁移并保持测试意图和覆盖率。

关键启发

  1. LLM coding 的核心是“生成 + 验证 + 反馈 + 重试”，不是单轮聊天。
  2. 选择相关上下文比微调提示词措辞更重要。
  3. 自动化迁移应 breadth-first 处理共性错误，最后再人工处理真正长尾。
