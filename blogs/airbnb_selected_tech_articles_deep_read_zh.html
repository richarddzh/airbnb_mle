二、Recommending Travel Destinations to Help Users Explore

原文：https://airbnb.tech/ai-ml/recommending-travel-destinations-to-help-users-explore/

业务问题

Airbnb 的一类用户处于旅行规划早期，并不清楚目的地、日期和偏好。他们可能只搜索“France”这类大范围地点，访问频率低、短期预订概率低，但如果能给出合适目的地推荐，就可能减少决策摩擦、提高 engagement 和 conversion。

核心技术

技术点具体做法为什么重要
行为序列 Transformer 把用户动作当作 token，输入 booking history、view history、search history。 同时捕捉短期兴趣（浏览/搜索）和长期偏好（历史预订）。
Action embedding 组合 每个 action 由 city、region、days-to-today embeddings 相加表示。 把地点与时间新鲜度编码进模型。
上下文特征 加入 current time 等信息。 捕捉季节性，例如夏季偏好、假期周期。
Active / dormant 双训练样本 每个 booking 构造 14 个样本：预订前 1-7 天模拟活跃用户；预订前 8-365 天随机采样且只用 booking data 模拟沉睡用户。 同时服务临近预订用户和早期探索/重新唤醒用户。
地理多任务学习 多预测头同时预测 region-level 和 city-level destination。 学习城市和区域之间的层级/邻近关系，例如 San Francisco 与 Bay Area。

上线场景与结果

模型用于两个产品：一是搜索框 autosuggest，在用户点击搜索框时推荐城市；二是 abandoned search email notification，用户放弃搜索后邮件推荐预测目的地区域的房源。文章提到在线 A/B 测试在非英语主导区域带来显著 booking gains，并且推荐不仅帮助未确定目的地的用户，也帮助愿意考虑邻近城市、更实惠房源的用户。

我的解读

这篇文章的亮点是将“语言模型的序列预测思想”迁移到旅行意图预测，但没有机械照搬 NLP。它针对 Airbnb 场景做了三处关键适配：训练样本同时覆盖 active/dormant 用户；动作 embedding 包含地理层级和时间距离；loss/architecture 引入城市与区域多任务一致性。这说明推荐系统的关键不是模型名，而是训练样本如何模拟真实决策阶段。

三、Improving Search Ranking for Maps

原文：https://airbnb.tech/ai-ml/improving-search-ranking-for-maps/

业务问题

Airbnb 搜索结果有 list-results 和 map-results 两种界面。传统排序假设用户注意力沿列表 rank 递减，因此按 booking probability 排序有效。但地图没有明确 rank，房源以 pin 散布在空间中，因此 list ranking 的注意力假设失效。

三阶段技术路线

阶段核心假设方法结果/观察
Uniform User Attention 地图 pin 获得近似均匀注意力。 限制展示 pin 中最高 booking probability 与最低 booking probability 的比值，避免太多低概率 pin 分散点击。 上线 restricted version 后获得 Airbnb ranking 历史中较大的 bookings improvement，并提升 5-star trip。
Tiered User Attention 不同视觉形态带来不同注意力。 高 booking probability 房源显示普通带价格 oval pins；较低概率房源显示 mini-pins。 mini-pins CTR 约为普通 pins 的 1/8，有助于桌面固定 pin 数量场景下分配注意力。
Discounted User Attention 地图中心/坐标位置影响注意力。 根据不同坐标的 CTR 热力，重新选择地图中心，使高 booking probability 房源更靠近注意力中心。 A/B 测试提升 uncancelled bookings 0.27%，map moves 减少 1.5%。

我的解读

这篇文章本质上是“界面形态改变排序目标”的案例。排序不是抽象地对 item 打分，而是要建模用户如何在当前 UI 中分配注意力。列表排序优化 rank；地图排序优化空间曝光、视觉层级和地图中心。这对推荐/搜索系统很重要：同一批候选，在不同呈现界面下需要不同 objective 和不同 evaluation metric。

五、Listening, Learning, and Helping at Scale

原文：https://airbnb.tech/ai-ml/listening-learning-and-helping-at-scale-how-machine-learning-transforms-airbnbs-voice-support-experience/

业务问题

传统 IVR 依赖按键菜单，用户体验生硬，且难以表达复杂问题。Airbnb 将 IVR 改为自然语言入口：用户说出问题，系统理解、检索帮助资源、发送解决方案或转人工。

端到端链路

  1. IVR 询问用户为何来电。
  2. Airbnb-specific ASR 将语音转文本。
  3. Contact Reason Detection 将问题分类到 refund、cancellation、account issue 等 taxonomy。
  4. 系统判断是否可 self-service；可解决则发送 Help Center article 或 workflow，不可解决或用户要求人工则转接坐席。
  5. Paraphrasing model 在发送文章前给用户解释系统理解到的问题。

关键技术与指标

模块技术关键点指标/效果
ASR从通用模型转向 noisy phone audio 适配，并用 Airbnb 领域短语优化。WER 从 33% 降至约 10%。
Intent DetectionContact Reason taxonomy + Issue Detection Service 并行部署。平均延迟低于 50ms。
Help Article RetrievalHelp article embedding 入向量库，cosine similarity 召回最多 30 篇。通常约 60ms。
LLM RankingLLM-based ranking model 对召回文章重排。以 Precision@N 持续评估。
ParaphrasingUX writers 预写标准 summary，embedding nearest-neighbor 匹配。人工评估 precision 超过 90%。

我的解读

这篇文章展示了一个务实的 AI 客服架构：并没有让 LLM 直接接管所有语音问题，而是将链路拆成 ASR、意图识别、检索、重排、解释、路由。尤其 paraphrasing 使用 curated summaries 而非完全自由生成，说明 Airbnb 在客服场景优先选择可控、可评估、低幻觉风险的方案。

六、Accelerating Large-Scale Test Migration with LLMs

原文：https://airbnb.tech/infrastructure/accelerating-large-scale-test-migration-with-llms/

业务问题

Airbnb 需要将近 3,500 个 React component test files 从 Enzyme 迁移到 React Testing Library。手工估计需要 1.5 年工程时间；直接删除会造成 coverage gaps；框架差异又不适合简单 find/replace。

LLM 迁移流水线

环节做法关键点
State machine每个文件经过 Enzyme refactor、fix Jest、fix lint、fix tsc、complete 等步骤。只有当前步骤验证通过才进入下一状态。
Retry loop验证失败时，把错误和最新文件版本回填给 LLM，多次重试直到通过或达到上限。对简单/中等复杂度文件非常有效，多数 10 次内完成。
Long context复杂文件 prompt 扩展到 40k-100k tokens，包含最多 50 个相关文件。主要收益来自选对相关文件，而不只是 prompt wording。
Progress stamping在文件中记录 MIGRATION STATUS JSON 注释。方便统计失败步骤、批量重跑和定位长尾问题。
Sample, tune, sweep抽样 5-10 个代表性失败文件，改 prompt/脚本，验证后扫全量。4 天从 75% 成功率提升到 97%。

结果

第一次 bulk run 4 小时自动迁移 75%；经过 4 天系统调优达到 97%；剩余 3% 由人工基于 LLM 生成 baseline 修复，一周完成。总计 6 周完成原本估计 1.5 年的迁移，并保持测试意图和 coverage。

我的解读

这篇文章的核心不是“LLM 会写代码”，而是“LLM 应该被放在验证驱动的自动化系统里”。Airbnb 把 LLM 当作可重试的 refactor worker，把 Jest/lint/tsc 当作判定器，把失败分布当作可优化对象。这比单次 prompt 更接近工业级 LLM coding workflow。

七、Embedding-Based Retrieval for Airbnb Search

原文：https://airbnb.tech/ai-ml/embedding-based-retrieval-for-airbnb-search/

业务问题

Airbnb 搜索需要从数百万房源中找出与 query 最相关的候选。大范围搜索如 California、France，或热门目的地如 Paris、London，会产生巨大候选集；Flexible date search 又进一步扩大了检索难度。EBR 的目标是在重排序模型之前，先高效缩小候选池。

核心技术

模块做法关键工程取舍
训练数据 按 trip 聚合用户历史 query；最终预订房源为 positive；曝光/互动但未预订房源为 hard negatives。 随机 negative 太容易，会导致模型效果差；hard negative 更贴近真实选择。
模型结构 Two-tower：listing tower 处理房源特征，query tower 处理地理、人数、住宿时长等 query 特征。 listing embeddings 可每日离线预计算，线上只算 query tower，降低延迟。
ANN 方案 比较 HNSW 与 IVF，最终选择 IVF。 HNSW recall 略好但面对价格/可订状态高频更新和地理 filter 延迟差；IVF 更好接入现有 search filter。
相似度函数 比较 dot product 与 Euclidean distance。 Euclidean distance 使 IVF cluster size 更均衡，避免大 cluster 降低判别力。

结果

EBR 完整上线到 Search 和 Email Marketing，在 A/B 测试中带来 statistically significant overall bookings gain，且提升幅度可与过去两年 Airbnb 搜索排序中最大级别的 ML 改进相当。核心收益来自更好地纳入 query context，在初始召回阶段就提升候选相关性。

我的解读

这篇文章体现了成熟搜索系统的典型分层：召回阶段追求规模和延迟，排序阶段再用更重模型。Airbnb 的关键工程判断是没有选择指标略优但线上代价高的 HNSW，而选择更适合高频更新、强过滤、易集成的 IVF。这说明生产系统中的“最佳算法”必须同时满足数据更新频率、过滤条件、内存、延迟和现有架构约束。

十、最终结论

这几篇文章共同说明，Airbnb 的 ML/AI 能力不是孤立的模型能力，而是围绕旅游双边市场构建的系统能力。对用户端，它用推荐和搜索降低探索成本；对客服端，它用语音 AI、检索和 LLM 改善自助支持与坐席效率；对工程端，它用 LLM 自动化迁移大规模遗留代码；对研究端，它把生产问题沉淀为 ranking、NLP、实验、优化和数据系统论文。

最值得借鉴的是 Airbnb 的工程范式：先用业务和界面定义正确问题，再构造贴近真实行为的数据，选择能满足线上约束的模型/索引/LLM 方案，最后用 A/B、验证器、人工反馈和业务指标闭环。
