深度解读：Recommending Travel Destinations to Help Users Explore

原文：Recommending Travel Destinations to Help Users Explore
本地参考：原文摘录 Markdown
作者：Weiwei Guo, Bin Xu, Sundara Rajan Srinivasavaradhan, Jie Tang, Xiaowei Liu, Bharathi Thangamani, Liwei He, Huiji Gao, Tracy Yu, Hui Gao, Stephanie Moyerman, Sanjeev Katariya
主题：目的地推荐、旅行探索、个性化、Transformer、行为序列建模、多任务学习、地理知识建模。

一句话总结

Airbnb 将用户在平台上的预订、浏览、搜索行为建模成类似语言 token 的序列，用 Transformer 预测用户下一步可能感兴趣的旅行目的地，并通过 active/dormant 用户样本构造与城市/区域多任务学习，让模型同时适配临近预订用户和早期探索用户。

业务背景：为什么需要目的地推荐

Airbnb 用户并不总是带着明确行程进入产品。很多用户处于 trip planning stage：他们可能不知道要去哪个城市、什么时候出发、预算多少、偏好哪类住宿。这类用户常搜索宽泛区域，例如 “France”，并且访问频率较低、短期预订概率较低。

这篇文章要解决的问题不是“用户输入城市后如何排序房源”，而是更上游的“用户还不知道去哪儿时，平台如何帮助他缩小选择范围”。这类推荐可以降低探索成本，激发灵感，并把模糊意图转化为可执行搜索。

挑战为什么难Airbnb 的应对
长期兴趣与短期兴趣并存 用户过去预订过 Hawaii 代表长期偏好，但最近搜索 Bay Area 代表当前计划。 同时输入 booking history、view history、search history 多类序列。
Active 与 dormant 用户行为差异大 活跃用户近期行为丰富，沉睡用户可能很久没访问，只有历史预订可用。 每个 booking 构造两组训练样本，分别模拟临近预订和早期规划。
地理关系复杂 城市、区域、邻近城市之间有层级和替代关系，例如 San Francisco 与 San Jose 都属于 Bay Area 语境。 通过 city-level 和 region-level 多任务学习注入地理知识。
旅行意图有季节性 同一个用户夏季和冬季可能偏好完全不同的目的地。 加入 current time 等上下文信息捕捉季节变化。
个性化需要隐私边界 目的地推荐依赖历史用户偏好数据。 文章明确提到用户可以 opt out of personalization。

模型架构解读

1. 把用户行为看作“语言 token”

文章说该框架 inspired by language modeling。这里的类比是：语言模型根据前面的词预测后续词；目的地推荐模型根据用户过去的旅行相关行为预测未来可能感兴趣的 destination。

Airbnb 将用户动作表示为 token，包括预订、浏览、搜索等。这些 token 不是自然语言，而是结构化行为事件。Transformer 负责从行为序列中学习兴趣演化。

2. Action 表示：city + region + days-to-today

每个 action 由多个 embedding 相加得到：

  • city embedding：表示具体城市偏好。
  • region embedding：表示更粗粒度地理区域偏好。
  • days-to-today embedding：表示行为距离当前时间多远，帮助模型区分近期兴趣和历史兴趣。

这种设计比只用城市 ID 更强，因为旅行目的地天然有层级结构和时间衰减。

3. 多序列融合

序列代表信号模型作用
Booking history真实完成预订的目的地。高置信长期偏好。
View history用户查看过的房源/目的地。中等强度兴趣，可能代表探索。
Search history用户主动搜索过的位置。强短期意图，尤其适合 active users。
Contextual time当前时间、季节。帮助模型理解季节性旅行需求。

补充分析：模型输入、输出与训练目标

维度文章信息解读
模型输入 用户在 Airbnb 上的 booking history、view history、search history，以及 current time 等上下文信息。 输入不是单一 query，而是多源用户行为序列。Booking 代表高置信长期偏好，view/search 代表近期探索和短期意图，当前时间用于捕捉季节性。
Action 表示 每个 action 表示为 city、region、days-to-today 三类 embedding 的和。 同一个用户行为同时携带地理粒度和时间距离信息：city/region 解决“去哪里”，days-to-today 解决“这个信号新不新”。
模型主体 使用 Transformer 对行为 token 序列建模。 可类比语言模型根据上下文预测后续 token；这里是根据历史旅行行为和上下文预测未来目的地意图。
模型输出 预测用户 destination intent，并在最后一层设置 city-level 与 region-level prediction heads。 输出可以是具体城市，也可以是更粗粒度区域。线上使用时可转化为 autosuggest 中的城市推荐，或 abandoned search 邮件里的目的地区域/房源推荐。
训练目标 根据 booking 前不同时间点可见的用户行为，预测该 booking 对应的目的地。 本质是监督式下一目的地预测。文章没有披露 loss 公式，但从架构看应包含 city 与 region 两个分类目标，并通过多任务学习让城市和区域预测保持地理一致性。

模型复杂度：公开信息与可推断范围

这确实是一个 基于 Transformer 的推荐模型。原文明确写到 “We use transformers to model sequences of user actions”，也就是用 Transformer 对 booking/view/search 等用户行为 token 序列建模。

但文章没有公开模型的具体复杂度：包括 Transformer 层数、hidden size、attention heads、参数量、序列长度上限、训练集 booking 总数、训练样本总量、训练资源和训练时长。因此不能严肃地给出“Airbnb 这个模型有多少层、多少参数”的确定答案。

问题原文是否披露可以确定什么不能确定什么
是否使用 Transformer 是 使用 Transformer 建模用户行为序列。 没有披露是 encoder-only、decoder-only，还是定制化 Transformer block。
层数 否 只能确认不是传统手工特征模型，而是序列深度模型。 无法确认是 2 层、4 层、6 层还是更深。
参数规模 否 参数主要来自 city/region/time embeddings、Transformer blocks、city/region prediction heads。 无法确认是百万级、千万级还是更大。
训练样本规模 部分披露 每个 booking 构造 14 条训练样本，因此样本数约为 14N，其中 N 是用于训练的 booking 数。 原文没有披露 N，所以无法得到绝对样本量。

从工程直觉看，这类目的地推荐 Transformer 通常不需要达到 LLM 的参数规模。原因是它的 token 不是自然语言词表，而是城市、区域、时间距离、行为类型等结构化特征；输出空间也是城市/区域目的地，而不是开放词表文本生成。因此它更可能是一个面向推荐排序/召回的中小型 Transformer，而不是百亿参数级大模型。

一个粗略的参数量拆解可以写成：

总参数量 ≈ embedding 参数 + Transformer block 参数 + 输出头参数
embedding 参数 ≈ (城市数 + 区域数 + 时间桶数 + 其他离散特征数) × hidden_size
Transformer 参数 ≈ 层数 × O(hidden_size²)
输出头参数 ≈ (城市类别数 + 区域类别数) × hidden_size

如果 hidden size、层数和城市/区域数量较小，模型可能是百万到千万参数级；如果目的地类别、embedding 维度和层数更大，也可能到数千万级。但这只是复杂度分析框架，不能当作 Airbnb 官方模型规格。

训练样本设计：文章最关键的工程点

Airbnb 为每个 booking 构造 14 个训练样本，分成两类：

样本类型构造方式模拟的用户状态可学习的信息
Active user 样本 从 booking date 前 1、2、3...7 天各构造一个样本。 用户已经接近预订，有近期搜索、浏览、可能已有明确目的地。 临近转化阶段的短期意图。
Dormant user 样本 从 booking date 前 8 到 365 天随机采样 7 个时间点，只使用 booking data。 用户处于早期规划，或者很久没访问 Airbnb。 长期旅行偏好与重新唤醒场景。

这比简单拿“预订前最后一次行为”训练更合理。因为上线场景中，模型既要服务已经活跃的用户，也要服务没有近期行为的沉睡用户。训练样本如果不模拟这两类状态，模型就可能只对短期强意图用户有效。

训练数据量口径

文章没有公开训练集中 booking 总数、用户数、时间跨度或最终样本总量，但披露了样本扩增规则：每个 booking 生成 14 条训练样本。因此如果原始 booking 数为 N，训练样本规模约为 14N。其中 7N 条用于模拟 active users，另外 7N 条用于模拟 dormant users。

样本组每个 booking 的样本数采样时间可用特征目标
Active user 7 booking 前 1、2、3、4、5、6、7 天 截至该时间点的 booking、view、search 数据 模拟临近预订阶段，学习短期强意图如何指向最终目的地。
Dormant user 7 booking 前 8 到 365 天随机采样 只使用 booking data 模拟早期规划或长时间未访问场景，学习长期偏好和季节性。

活跃用户与休眠用户的差异化处理

这篇文章的一个关键点是：Airbnb 没有用同一套样本假设覆盖所有用户，而是显式把用户状态拆成 active 和 dormant 两种训练视角。

用户类型典型线上状态训练时如何模拟模型应重点学习推荐形态
Active users 最近有搜索、浏览或其他站内行为，通常已接近预订决策。 使用 booking 前 1-7 天的样本，并保留当时可见的 booking/view/search 数据。 近期搜索和浏览信号。例如用户上周搜索 California Bay Area，模型应理解其当前可能在比较 Bay Area 及附近更实惠城市。 更适合在 autosuggest 中给出具体城市或邻近城市，帮助用户快速进入搜索结果。
Dormant users 很久没访问 Airbnb，缺少近期搜索和浏览行为，只能依赖历史预订。 从 booking 前 8-365 天随机采样，并只输入 booking history。 长期旅行偏好、历史目的地类型和季节性。例如过去常在夏季预订海滨或度假型目的地。 更适合在召回邮件、灵感推荐等场景中给出较宽泛目的地，重新激活用户探索。

地理理解：多任务学习

Airbnb 拥有丰富地理知识：城市、区域、邻近城市、替代目的地之间存在关系。文章以 Bay Area 为例：对 San Francisco 感兴趣的用户，也可能考虑 San Jose 等附近城市。

为此，模型在最后一层增加多个 prediction heads，同时预测 region-level 和 city-level destination。这样做有两个好处：

  1. 让模型学习地理层级：城市属于区域，区域包含多个可替代城市。
  2. 提升推荐泛化：当用户意图模糊时，可以先预测区域；当意图明确时，可以推荐具体城市。

这属于对地理位置信息的特殊处理：目的地不是扁平类别，城市之间也不是互斥独立关系。将 region 作为显式预测目标，可以把 San Francisco、San Jose 等城市放在 Bay Area 这个共同语义空间下学习；当用户对某个城市感兴趣时，模型更容易扩展到同区域的替代城市，而不是只机械复现历史城市。

地理信息处理方式解决的问题推荐收益
action embedding 同时包含 city 与 region 单个城市 ID 无法表达所属区域和相邻关系。 让用户历史行为同时贡献精细城市偏好和粗粒度区域偏好。
最后一层同时预测 city-level 与 region-level destination 只预测城市容易过窄，只预测区域又不够可执行。 模型既能推荐具体城市，也能在意图模糊时保留区域级泛化能力。
鼓励 city/region 预测一致 城市和区域预测若彼此冲突，会降低推荐可信度。 帮助模型学习更稳定的地理表示，支持邻近城市和替代目的地推荐。

产品场景如何使用模型业务意义
Autosuggest 用户点击搜索框时，展示多个城市推荐。 在用户尚未输入具体目的地前提供灵感，缩短搜索路径。

文章提到，在线 A/B 测试在非英语主导区域带来显著 booking gains；进一步分析显示，该推荐不仅帮助未决定目的地的用户，也帮助愿意考虑邻近城市、更实惠房源的用户。

模型使用效果

文章披露的是业务层面的线上效果，而不是离线指标。模型已落地到两个触点，并通过 A/B 测试验证对 booking 有正向影响。

应用场景模型如何使用文章披露的效果分析

可复用方法论

  1. 把行为序列语言化：很多非文本问题可以用 language modeling 思路处理，但 token、上下文和目标要按业务重新定义。
  2. 训练样本要覆盖上线用户状态：如果线上有 active 和 dormant 两类用户，训练集也必须显式模拟这两类状态。
  3. 利用结构化领域知识：旅行目的地不是扁平类别，城市/区域/邻近关系应进入模型。
  4. 一个 intent model 可服务多个触点：搜索 autosuggest 和邮件召回共享目的地意图模型，提高模型复用价值。
  5. 评价应看最终业务指标：文章强调 A/B 测试和 booking gains，而不只是离线准确率。

潜在局限与开放问题

  1. 文章没有披露具体 loss function、负采样策略、离线指标和线上 A/B 数值。
  2. 目的地推荐可能影响用户探索多样性，需要平衡个性化与新颖性。
  3. 非英语区域效果显著，但不同文化和市场的旅行偏好可能需要本地化评估。
  4. 用户可 opt out personalization，但文章未展开隐私保护、数据保留和模型训练排除机制。

最终结论

这篇文章的核心价值在于训练数据设计和地理知识建模。Airbnb 没有只做一个“下一目的地分类器”，而是围绕旅行规划的真实阶段构造样本，用 Transformer 融合多类行为序列，再通过区域/城市多任务学习增强地理理解。它是 Airbnb 从搜索平台走向 AI travel planning 的关键基础模块。
