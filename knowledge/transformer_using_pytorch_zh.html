重点问题：split_heads 里的 transpose(1, 2) 是什么作用？

文章中的代码如下：

def split_heads(self, x):
    batch_size, seq_length, d_model = x.size()
    return x.view(batch_size, seq_length, self.num_heads, self.d_k).transpose(1, 2)

假设输入 x 的 shape 是：

(batch_size, seq_length, d_model)

其中：

  • batch_size：一次输入多少个样本。
  • seq_length：序列长度，比如一句话有多少个 token。
  • d_model：每个 token 的 embedding 维度。

Multi-Head Attention 会把 d_model 拆成多个 head：

d_model = num_heads * d_k

所以这一步：

x.view(batch_size, seq_length, self.num_heads, self.d_k)

会把 shape 从：

(batch_size, seq_length, d_model)

变成：

(batch_size, seq_length, num_heads, d_k)

也就是：每个 token 的大向量被拆成了 num_heads 个小向量。

但是这个 shape 还不适合后面计算 attention。Scaled Dot-Product Attention 通常希望每个 head 独立计算注意力，因此更方便的 shape 是：

(batch_size, num_heads, seq_length, d_k)
transpose(1, 2) 的作用就是交换第 1 维和第 2 维，把张量从“按 token 组织 head”改成“按 head 组织完整序列”。

在 PyTorch 中维度编号从 0 开始：

原始 view 后：
0: batch_size
1: seq_length
2: num_heads
3: d_k

执行 transpose(1, 2) 后变成：

0: batch_size
1: num_heads
2: seq_length
3: d_k

也就是：

(batch_size, seq_length, num_heads, d_k)
        ↓ transpose(1, 2)
(batch_size, num_heads, seq_length, d_k)

为什么必须这样换维度？

因为后面的 attention 计算是：

attn_scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(self.d_k)

此时如果 Q 和 K 的 shape 是：

Q: (batch_size, num_heads, seq_length, d_k)
K: (batch_size, num_heads, seq_length, d_k)

那么：

K.transpose(-2, -1)

会把 K 的最后两维交换：

K:                  (batch_size, num_heads, seq_length, d_k)
K.transpose(-2,-1): (batch_size, num_heads, d_k, seq_length)

于是矩阵乘法变成：

(seq_length, d_k) @ (d_k, seq_length) = (seq_length, seq_length)

所以 attn_scores 的 shape 是：

(batch_size, num_heads, seq_length, seq_length)

含义是：对每个 batch、每个 head，序列中每个 token 都会对所有 token 计算一个 attention score。

如果不做 transpose(1, 2)，shape 会停留在：

(batch_size, seq_length, num_heads, d_k)

这时 PyTorch 会把 seq_length 当成一个批处理维度，把 num_heads 放在矩阵乘法的倒数第二维，语义就错了。你想要的是“每个 head 独立对整条序列做 attention”，而不是“每个位置拿 head 维度去做 attention”。

一个具体例子

假设：

batch_size = 2
seq_length = 5
d_model = 512
num_heads = 8
d_k = 64

输入：

x.shape = (2, 5, 512)

先拆 head：

x.view(2, 5, 8, 64)

得到：

(2, 5, 8, 64)

含义是：2 个样本，每个样本 5 个 token，每个 token 被拆成 8 个 head，每个 head 64 维。

再交换维度：

x.transpose(1, 2)

得到：

(2, 8, 5, 64)

含义变成：2 个样本，每个样本 8 个 head，每个 head 都有一条长度为 5 的序列，每个 token 在这个 head 下是 64 维。

这个形状正好适合并行计算 8 个 attention head。

直觉理解

transpose(1, 2) 的目的不是改变数据本身的数值，而是改变数据的组织方式。

原来：按 token 看，每个 token 里面有多个 head
变成：按 head 看，每个 head 里面有完整的 token 序列

Multi-Head Attention 的目标是让不同 head 从不同表示子空间观察整句话。因此每个 head 都必须拿到完整的序列，而不是只拿到某个 token 的一部分。

和 combine_heads 的关系

文章中还有反向操作：

def combine_heads(self, x):
    batch_size, _, seq_length, d_k = x.size()
    return x.transpose(1, 2).contiguous().view(batch_size, seq_length, self.d_model)

attention 计算完成后，x 的 shape 是：

(batch_size, num_heads, seq_length, d_k)

先执行：

x.transpose(1, 2)

变回：

(batch_size, seq_length, num_heads, d_k)

再通过：

view(batch_size, seq_length, self.d_model)

把多个 head 拼回原来的 d_model 维度：

(batch_size, seq_length, d_model)

这里的 .contiguous() 很重要，因为 transpose 通常只改变张量的 stride 视图，不一定让内存连续排列；而 view 要求张量在内存中是连续的。没有 .contiguous() 时，后面的 view 可能报错或行为不符合预期。

补充问题一：nn.Linear 总是在最后一个维度上计算吗？

是的。PyTorch 中的 nn.Linear 会作用在输入 tensor 的最后一个维度上，要求最后一维等于 in_features，然后把最后一维变成 out_features。前面的维度都会原样保留，可以理解成 batch-like 维度。

一般形式是：

输入:  (*, in_features)
输出:  (*, out_features)

其中 * 表示任意数量的前置维度，比如 batch、sequence、head 等。

例如：

linear = nn.Linear(512, 512)
x.shape = (batch_size, seq_length, 512)
y = linear(x)

输出 shape 是：

y.shape = (batch_size, seq_length, 512)

它等价于对每个样本、每个 token 的最后一维向量单独做一次 512 -> 512 的线性投影。

所以在 Transformer 里：

Q = self.W_q(Q)
K = self.W_k(K)
V = self.W_v(V)

如果输入是：

(batch_size, seq_length, d_model)

那么 nn.Linear(d_model, d_model) 会对每个 token 的 d_model 向量做投影，输出仍然是：

(batch_size, seq_length, d_model)
直觉上，nn.Linear 不关心前面有多少维；它只看最后一维是不是自己的 in_features。

补充问题二：attn_scores 里的 K.transpose(-2, -1) 是什么意思？

在这行代码里：

attn_scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(self.d_k)

K.transpose(-2, -1) 的作用是交换 K 的最后两个维度。这里的 -1 表示最后一维，-2 表示倒数第二维。

假设：

Q.shape = (batch_size, num_heads, seq_length, d_k)
K.shape = (batch_size, num_heads, seq_length, d_k)

那么：

K.transpose(-2, -1)

会把 K 从：

(batch_size, num_heads, seq_length, d_k)

变成：

(batch_size, num_heads, d_k, seq_length)

这样矩阵乘法就变成了 attention 公式里的 QK^T：

(seq_length, d_k) @ (d_k, seq_length)
= (seq_length, seq_length)

最终得到：

attn_scores.shape = (batch_size, num_heads, seq_length, seq_length)

它的含义是：每个 token 的 Query 都会和所有 token 的 Key 做点积，得到“这个 token 应该关注其它每个 token 多少”的分数。

注意区分两个 transpose：

位置 代码 作用
split_heads transpose(1, 2) 把 head 维度提前，从而让每个 head 拿到完整序列。
attn_scores K.transpose(-2, -1) 把 Key 的最后两维转置，实现矩阵乘法里的 K^T。

补充问题三：高维 tensor 做 torch.matmul 时，只计算后两维吗？

是的。当 torch.matmul 的输入是高维 tensor 时，它会把最后两维当作矩阵维度做乘法，前面的维度当作 batch 维度。

一般形式是：

A.shape = (..., m, n)
B.shape = (..., n, p)

torch.matmul(A, B).shape = (..., m, p)

其中 ... 是前置 batch 维度，并且这些 batch 维度需要相同或可以广播。

在 Multi-Head Attention 中：

A.shape = (batch_size, num_heads, seq_length, d_k)
B.shape = (batch_size, num_heads, d_k, seq_length)

torch.matmul(A, B) 实际做的是：

(seq_length, d_k) @ (d_k, seq_length)

前面的：

(batch_size, num_heads)

只是 batch 维度，表示对每个 batch、每个 head 都独立做一次矩阵乘法。

所以输出是：

(batch_size, num_heads, seq_length, seq_length)

torch.matmul 的前置 batch 维度还支持 broadcasting。例如：

A.shape = (2, 8, 5, 64)
B.shape = (1, 8, 64, 5)

这里 batch 维度 (2, 8) 和 (1, 8) 可以广播成 (2, 8)，所以它们也可以相乘。

补充问题四：为什么 masked_fill 不填 0，而是填 -1e9？

相关代码是：

if mask is not None:
    attn_scores = attn_scores.masked_fill(mask == 0, -1e9)

attn_probs = torch.softmax(attn_scores, dim=-1)

关键点是：masked_fill 发生在 softmax 之前。此时 attn_scores 还不是概率，而是 logits，也就是还没有归一化的注意力分数。

softmax 的形式是：

softmax(x_i) = exp(x_i) / sum(exp(x_j))

如果把被 mask 的位置填成 0，它仍然会参与 softmax，因为 exp(0) = 1，概率可能并不小。例如：

scores = [2, 1, 0]
softmax(scores) ≈ [0.665, 0.245, 0.090]

如果第三个位置本来应该被 mask，但你填了 0，它仍然拿到了约 9% 的注意力权重。

-1e9 表示：

-1e9 = -1 * 10^9 = -1000000000

这是一个极大的负数。因为：

exp(-1e9) ≈ 0

所以 softmax 后，被 mask 的位置概率几乎就是 0。

填 0 只是给它一个普通分数；填 -1e9 等价于告诉 softmax：“这个位置绝对不要看”。

实际工程中也常见 float("-inf") 或 dtype 对应的最小值。教学代码里常用 -1e9，因为简单、直观，数值上足够接近负无穷。

补充问题五：softmax(attn_scores, dim=-1) 在哪个维度上做？结果是什么形状？

在 Multi-Head Attention 中，attn_scores 的形状通常是：

(batch_size, num_heads, seq_length, seq_length)

更一般地写是：

(batch_size, num_heads, query_len, key_len)

含义是：

每个 batch
  每个 head
    每个 query token
      对每个 key token 的注意力分数

这句代码：

attn_probs = torch.softmax(attn_scores, dim=-1)

dim=-1 表示在最后一维上做 softmax，也就是在 key_len 这个维度上归一化。

换句话说，对每个 query token 来说，它会把“这个 query 对所有 key 的分数”归一化成概率：

attn_scores[b, h, q, :] -> softmax

其中：

attn_scores[b, h, q, :].shape = (key_len,)
attn_probs[b, h, q, :].shape  = (key_len,)

并且这一行概率和为 1：

sum(attn_probs[b, h, q, :]) = 1

softmax 不改变 tensor 的整体形状，所以结果仍然是：

attn_probs.shape = (batch_size, num_heads, query_len, key_len)

在 self-attention 中通常 query_len == key_len == seq_length，所以就是：

(batch_size, num_heads, seq_length, seq_length)
直觉上：每个 token 作为 query 时，对整句话里所有 token 分配一组注意力权重。

补充问题六：contiguous() 是什么作用？

在 combine_heads 中：

def combine_heads(self, x):
    batch_size, _, seq_length, d_k = x.size()
    return x.transpose(1, 2).contiguous().view(batch_size, seq_length, self.d_model)

contiguous() 的作用是把一个内存不连续的 tensor 重新整理成内存连续的 tensor，这样后面的 view(...) 才能安全工作。

transpose(1, 2) 会把 shape 从：

(batch_size, num_heads, seq_length, d_k)

变成：

(batch_size, seq_length, num_heads, d_k)

但是 PyTorch 的 transpose 通常不会真的复制数据并重新排列内存，它只是改变 tensor 的视图和 stride。因此，看起来维度换了，但底层内存排列可能还是原来的顺序。

而 view(...) 要求 tensor 在内存中是连续排列的，否则它不知道怎么按照新 shape 正确解释内存。所以这里需要：

x.transpose(1, 2).contiguous().view(batch_size, seq_length, self.d_model)

把：

(batch_size, seq_length, num_heads, d_k)

合并成：

(batch_size, seq_length, d_model)

其中：

d_model = num_heads * d_k
transpose 改变维度视图，但可能导致内存不连续；contiguous 重新整理内存；view 依赖连续内存来 reshape。

现在很多时候也可以写成：

x.transpose(1, 2).reshape(batch_size, seq_length, self.d_model)

因为 reshape 在需要时会自动复制成连续内存。但教学代码里经常写成 transpose(...).contiguous().view(...)，这样更明确地表达：先换维度，再保证内存连续，再合并 head。

补充问题七：PyTorch 有没有 SwiGLU？如何用 SwiGLU 实现 FFN？

PyTorch 标准库里有 nn.GLU，也有 F.silu / nn.SiLU，但通常没有一个专门叫 nn.SwiGLU 的模块。SwiGLU 一般自己用两个 Linear 加 SiLU 实现。

经典 Transformer FFN 是：

FFN(x) = W2(activation(W1(x)))

SwiGLU FFN 是：

FFN(x) = W_down(SiLU(W_gate(x)) * W_up(x))

也就是先做两个并行投影：一个产生 gate，一个产生 value，然后用 SiLU(gate) * value 做逐元素门控。

import torch
from torch import nn
import torch.nn.functional as F

class SwiGLUFFN(nn.Module):
    def __init__(self, d_model, d_ff):
        super().__init__()
        self.w_gate = nn.Linear(d_model, d_ff)
        self.w_up = nn.Linear(d_model, d_ff)
        self.w_down = nn.Linear(d_ff, d_model)

    def forward(self, x):
        return self.w_down(F.silu(self.w_gate(x)) * self.w_up(x))

如果原来的 FFN 是：

class PositionWiseFeedForward(nn.Module):
    def __init__(self, d_model, d_ff):
        super().__init__()
        self.fc1 = nn.Linear(d_model, d_ff)
        self.fc2 = nn.Linear(d_ff, d_model)
        self.relu = nn.ReLU()

    def forward(self, x):
        return self.fc2(self.relu(self.fc1(x)))

可以替换成：

class PositionWiseSwiGLUFeedForward(nn.Module):
    def __init__(self, d_model, d_ff):
        super().__init__()
        self.gate_proj = nn.Linear(d_model, d_ff)
        self.up_proj = nn.Linear(d_model, d_ff)
        self.down_proj = nn.Linear(d_ff, d_model)

    def forward(self, x):
        x = F.silu(self.gate_proj(x)) * self.up_proj(x)
        return self.down_proj(x)

输入如果是：

(batch_size, seq_length, d_model)

那么中间 shape 是：

gate_proj(x): (batch_size, seq_length, d_ff)
up_proj(x):   (batch_size, seq_length, d_ff)
相乘后:       (batch_size, seq_length, d_ff)
down_proj:    (batch_size, seq_length, d_model)

注意参数量：SwiGLU 有三个 Linear，而普通 FFN 只有两个 Linear。所以很多 LLaMA 类模型会把 SwiGLU 的 hidden dim 调小，比如用 int(2/3 * 4 * d_model)，这样参数量和普通 4 * d_model FFN 接近。

补充问题八：PyTorch 的 SiLU 计算是什么？

torch.nn.SiLU / torch.nn.functional.silu 计算的是 Sigmoid Linear Unit，也叫 Swish。

公式是：

SiLU(x) = x * sigmoid(x)

其中：

sigmoid(x) = 1 / (1 + exp(-x))

所以：

SiLU(x) = x / (1 + exp(-x))

PyTorch 中：

import torch
import torch.nn.functional as F

y = F.silu(x)

等价于：

y = x * torch.sigmoid(x)

它和 ReLU 的区别是：ReLU 对负数直接截断成 0，而 SiLU 对负数是平滑压小，不是硬截断。

ReLU(x) = max(0, x)
SiLU(x) = x * sigmoid(x)

在 SwiGLU 里：

F.silu(gate_proj(x)) * up_proj(x)

意思是先用 SiLU 生成一个平滑的 gate，再用这个 gate 去逐元素调制另一条分支。

补充问题九：这是 Pre-LN 还是 Post-LN？RMSNorm 如何计算？

文章中的 DecoderLayer 代码是：

def forward(self, x, enc_output, src_mask, tgt_mask):
    attn_output = self.self_attn(x, x, x, tgt_mask)
    x = self.norm1(x + self.dropout(attn_output))
    attn_output = self.cross_attn(x, enc_output, enc_output, src_mask)
    x = self.norm2(x + self.dropout(attn_output))
    ff_output = self.feed_forward(x)
    x = self.norm3(x + self.dropout(ff_output))
    return x

这是 Post-LayerNorm（Post-LN），因为它的结构是：

x = LayerNorm(x + Sublayer(x))

也就是先做子层计算和残差连接，再做 LayerNorm。LayerNorm 放在残差连接之后，所以叫 Post-LN。

如果是 Pre-LayerNorm（Pre-LN），通常会写成：

norm_x = self.norm1(x)
attn_output = self.self_attn(norm_x, norm_x, norm_x, tgt_mask)
x = x + self.dropout(attn_output)

结构是：

x = x + Sublayer(LayerNorm(x))

也就是 LayerNorm 放在子层之前。

结构 公式 特点
Post-LN LayerNorm(x + Sublayer(x)) 原始 Transformer 常用；深层模型训练更容易不稳定。
Pre-LN x + Sublayer(Norm(x)) 现代深层 Transformer / LLM 更常用；梯度更容易沿残差路径传播。

现代主流 LLM 基本采用 Pre-LN，并且很多模型不用标准 LayerNorm，而使用 RMSNorm。LLaMA 风格大致是：

h = x + self_attn(rms_norm(x))
out = h + mlp(rms_norm(h))

RMSNorm 如何计算？

RMSNorm 的核心是：只用均方根 RMS 做缩放，不减均值。

给定输入向量：

x = [x1, x2, ..., xd]

RMS 是：

RMS(x) = sqrt((1/d) * sum(x_i^2))

RMSNorm 计算：

RMSNorm(x) = x / sqrt(mean(x^2) + eps) * weight

也就是：

y_i = x_i / sqrt((1/d) * sum_j(x_j^2) + eps) * g_i

其中 g_i 是可学习缩放参数 weight，eps 是防止除 0 的小常数。

PyTorch 实现大概是：

class RMSNorm(nn.Module):
    def __init__(self, dim, eps=1e-6):
        super().__init__()
        self.weight = nn.Parameter(torch.ones(dim))
        self.eps = eps

    def forward(self, x):
        rms = torch.sqrt(torch.mean(x * x, dim=-1, keepdim=True) + self.eps)
        x = x / rms
        return self.weight * x

也常写成更高效的形式：

x = x * torch.rsqrt(x.pow(2).mean(dim=-1, keepdim=True) + eps)

其中 rsqrt(z) 表示：

rsqrt(z) = 1 / sqrt(z)

LayerNorm 和 RMSNorm 的区别：

Norm 计算方式 是否减均值
LayerNorm (x - mean(x)) / sqrt(var(x) + eps) 是
RMSNorm x / sqrt(mean(x^2) + eps) 否
总结：文章代码是 Post-LN；现代主流 LLM 多用 Pre-LN，并常用 RMSNorm 替代 LayerNorm，以提升深层模型训练稳定性并减少一点计算。

知识点 为什么重要
Tensor shape 深度学习工程中大量错误来自维度不匹配。
view / reshape 用来改变张量形状，常用于拆分或合并维度。
transpose 用来交换维度，使张量符合矩阵乘法或广播需求。
Batched matrix multiplication Transformer attention 依赖高维批量矩阵乘法。
Attention score shape 通常是 (batch_size, num_heads, query_len, key_len)。
Memory layout / contiguous 理解 PyTorch 中 view 和 transpose 的底层约束。
Mask before softmax 理解为什么用极大负数屏蔽 token，而不是把 score 设为 0。
softmax(dim=-1) 理解 attention 中每个 query 对所有 key 的权重归一化。
SwiGLU / SiLU 理解现代 LLM FFN 常用的门控激活结构。
Pre-LN / Post-LN / RMSNorm 理解现代深层 Transformer 为什么通常把 norm 放到子层之前，并常用 RMSNorm。

总结

split_heads 里的 transpose(1, 2) 是为了把张量从：

(batch_size, seq_length, num_heads, d_k)

转换为：

(batch_size, num_heads, seq_length, d_k)

这样每个 attention head 都可以独立地对完整序列计算 self-attention。它是 Multi-Head Attention 能正确并行计算多个 head 的关键步骤。
