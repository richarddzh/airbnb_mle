重点问题：split_heads 里的 transpose(1, 2) 是什么作用？

文章中的代码如下：

def split_heads(self, x):
    batch_size, seq_length, d_model = x.size()
    return x.view(batch_size, seq_length, self.num_heads, self.d_k).transpose(1, 2)

假设输入 x 的 shape 是：

(batch_size, seq_length, d_model)

其中：

  • batch_size：一次输入多少个样本。
  • seq_length：序列长度，比如一句话有多少个 token。
  • d_model：每个 token 的 embedding 维度。

Multi-Head Attention 会把 d_model 拆成多个 head：

d_model = num_heads * d_k

所以这一步：

x.view(batch_size, seq_length, self.num_heads, self.d_k)

会把 shape 从：

(batch_size, seq_length, d_model)

变成：

(batch_size, seq_length, num_heads, d_k)

也就是：每个 token 的大向量被拆成了 num_heads 个小向量。

但是这个 shape 还不适合后面计算 attention。Scaled Dot-Product Attention 通常希望每个 head 独立计算注意力，因此更方便的 shape 是：

(batch_size, num_heads, seq_length, d_k)
transpose(1, 2) 的作用就是交换第 1 维和第 2 维，把张量从“按 token 组织 head”改成“按 head 组织完整序列”。

在 PyTorch 中维度编号从 0 开始：

原始 view 后：
0: batch_size
1: seq_length
2: num_heads
3: d_k

执行 transpose(1, 2) 后变成：

0: batch_size
1: num_heads
2: seq_length
3: d_k

也就是：

(batch_size, seq_length, num_heads, d_k)
        ↓ transpose(1, 2)
(batch_size, num_heads, seq_length, d_k)

为什么必须这样换维度？

因为后面的 attention 计算是：

attn_scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(self.d_k)

此时如果 Q 和 K 的 shape 是：

Q: (batch_size, num_heads, seq_length, d_k)
K: (batch_size, num_heads, seq_length, d_k)

那么：

K.transpose(-2, -1)

会把 K 的最后两维交换：

K:                  (batch_size, num_heads, seq_length, d_k)
K.transpose(-2,-1): (batch_size, num_heads, d_k, seq_length)

于是矩阵乘法变成：

(seq_length, d_k) @ (d_k, seq_length) = (seq_length, seq_length)

所以 attn_scores 的 shape 是：

(batch_size, num_heads, seq_length, seq_length)

含义是：对每个 batch、每个 head，序列中每个 token 都会对所有 token 计算一个 attention score。

如果不做 transpose(1, 2)，shape 会停留在：

(batch_size, seq_length, num_heads, d_k)

这时 PyTorch 会把 seq_length 当成一个批处理维度，把 num_heads 放在矩阵乘法的倒数第二维，语义就错了。你想要的是“每个 head 独立对整条序列做 attention”，而不是“每个位置拿 head 维度去做 attention”。

一个具体例子

假设：

batch_size = 2
seq_length = 5
d_model = 512
num_heads = 8
d_k = 64

输入：

x.shape = (2, 5, 512)

先拆 head：

x.view(2, 5, 8, 64)

得到：

(2, 5, 8, 64)

含义是：2 个样本，每个样本 5 个 token，每个 token 被拆成 8 个 head，每个 head 64 维。

再交换维度：

x.transpose(1, 2)

得到：

(2, 8, 5, 64)

含义变成：2 个样本，每个样本 8 个 head，每个 head 都有一条长度为 5 的序列，每个 token 在这个 head 下是 64 维。

这个形状正好适合并行计算 8 个 attention head。

直觉理解

transpose(1, 2) 的目的不是改变数据本身的数值，而是改变数据的组织方式。

原来：按 token 看，每个 token 里面有多个 head
变成：按 head 看，每个 head 里面有完整的 token 序列

Multi-Head Attention 的目标是让不同 head 从不同表示子空间观察整句话。因此每个 head 都必须拿到完整的序列，而不是只拿到某个 token 的一部分。

和 combine_heads 的关系

文章中还有反向操作：

def combine_heads(self, x):
    batch_size, _, seq_length, d_k = x.size()
    return x.transpose(1, 2).contiguous().view(batch_size, seq_length, self.d_model)

attention 计算完成后，x 的 shape 是：

(batch_size, num_heads, seq_length, d_k)

先执行：

x.transpose(1, 2)

变回：

(batch_size, seq_length, num_heads, d_k)

再通过：

view(batch_size, seq_length, self.d_model)

把多个 head 拼回原来的 d_model 维度：

(batch_size, seq_length, d_model)

这里的 .contiguous() 很重要，因为 transpose 通常只改变张量的 stride 视图，不一定让内存连续排列；而 view 要求张量在内存中是连续的。没有 .contiguous() 时，后面的 view 可能报错或行为不符合预期。

补充问题一：nn.Linear 总是在最后一个维度上计算吗？

是的。PyTorch 中的 nn.Linear 会作用在输入 tensor 的最后一个维度上，要求最后一维等于 in_features，然后把最后一维变成 out_features。前面的维度都会原样保留，可以理解成 batch-like 维度。

一般形式是：

输入:  (*, in_features)
输出:  (*, out_features)

其中 * 表示任意数量的前置维度，比如 batch、sequence、head 等。

例如：

linear = nn.Linear(512, 512)
x.shape = (batch_size, seq_length, 512)
y = linear(x)

输出 shape 是：

y.shape = (batch_size, seq_length, 512)

它等价于对每个样本、每个 token 的最后一维向量单独做一次 512 -> 512 的线性投影。

所以在 Transformer 里：

Q = self.W_q(Q)
K = self.W_k(K)
V = self.W_v(V)

如果输入是：

(batch_size, seq_length, d_model)

那么 nn.Linear(d_model, d_model) 会对每个 token 的 d_model 向量做投影，输出仍然是：

(batch_size, seq_length, d_model)
直觉上，nn.Linear 不关心前面有多少维；它只看最后一维是不是自己的 in_features。

补充问题二：attn_scores 里的 K.transpose(-2, -1) 是什么意思？

在这行代码里：

attn_scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(self.d_k)

K.transpose(-2, -1) 的作用是交换 K 的最后两个维度。这里的 -1 表示最后一维，-2 表示倒数第二维。

假设：

Q.shape = (batch_size, num_heads, seq_length, d_k)
K.shape = (batch_size, num_heads, seq_length, d_k)

那么：

K.transpose(-2, -1)

会把 K 从：

(batch_size, num_heads, seq_length, d_k)

变成：

(batch_size, num_heads, d_k, seq_length)

这样矩阵乘法就变成了 attention 公式里的 QK^T：

(seq_length, d_k) @ (d_k, seq_length)
= (seq_length, seq_length)

最终得到：

attn_scores.shape = (batch_size, num_heads, seq_length, seq_length)

它的含义是：每个 token 的 Query 都会和所有 token 的 Key 做点积，得到“这个 token 应该关注其它每个 token 多少”的分数。

注意区分两个 transpose：

位置 代码 作用
split_heads transpose(1, 2) 把 head 维度提前，从而让每个 head 拿到完整序列。
attn_scores K.transpose(-2, -1) 把 Key 的最后两维转置，实现矩阵乘法里的 K^T。

补充问题三：高维 tensor 做 torch.matmul 时，只计算后两维吗？

是的。当 torch.matmul 的输入是高维 tensor 时，它会把最后两维当作矩阵维度做乘法，前面的维度当作 batch 维度。

一般形式是：

A.shape = (..., m, n)
B.shape = (..., n, p)

torch.matmul(A, B).shape = (..., m, p)

其中 ... 是前置 batch 维度，并且这些 batch 维度需要相同或可以广播。

在 Multi-Head Attention 中：

A.shape = (batch_size, num_heads, seq_length, d_k)
B.shape = (batch_size, num_heads, d_k, seq_length)

torch.matmul(A, B) 实际做的是：

(seq_length, d_k) @ (d_k, seq_length)

前面的：

(batch_size, num_heads)

只是 batch 维度，表示对每个 batch、每个 head 都独立做一次矩阵乘法。

所以输出是：

(batch_size, num_heads, seq_length, seq_length)

torch.matmul 的前置 batch 维度还支持 broadcasting。例如：

A.shape = (2, 8, 5, 64)
B.shape = (1, 8, 64, 5)

这里 batch 维度 (2, 8) 和 (1, 8) 可以广播成 (2, 8)，所以它们也可以相乘。

补充问题四：为什么 masked_fill 不填 0，而是填 -1e9？

相关代码是：

if mask is not None:
    attn_scores = attn_scores.masked_fill(mask == 0, -1e9)

attn_probs = torch.softmax(attn_scores, dim=-1)

关键点是：masked_fill 发生在 softmax 之前。此时 attn_scores 还不是概率，而是 logits，也就是还没有归一化的注意力分数。

softmax 的形式是：

softmax(x_i) = exp(x_i) / sum(exp(x_j))

如果把被 mask 的位置填成 0，它仍然会参与 softmax，因为 exp(0) = 1，概率可能并不小。例如：

scores = [2, 1, 0]
softmax(scores) ≈ [0.665, 0.245, 0.090]

如果第三个位置本来应该被 mask，但你填了 0，它仍然拿到了约 9% 的注意力权重。

-1e9 表示：

-1e9 = -1 * 10^9 = -1000000000

这是一个极大的负数。因为：

exp(-1e9) ≈ 0

所以 softmax 后，被 mask 的位置概率几乎就是 0。

填 0 只是给它一个普通分数；填 -1e9 等价于告诉 softmax：“这个位置绝对不要看”。

实际工程中也常见 float("-inf") 或 dtype 对应的最小值。教学代码里常用 -1e9，因为简单、直观，数值上足够接近负无穷。

补充问题五：softmax(attn_scores, dim=-1) 在哪个维度上做？结果是什么形状？

在 Multi-Head Attention 中，attn_scores 的形状通常是：

(batch_size, num_heads, seq_length, seq_length)

更一般地写是：

(batch_size, num_heads, query_len, key_len)

含义是：

每个 batch
  每个 head
    每个 query token
      对每个 key token 的注意力分数

这句代码：

attn_probs = torch.softmax(attn_scores, dim=-1)

dim=-1 表示在最后一维上做 softmax，也就是在 key_len 这个维度上归一化。

换句话说，对每个 query token 来说，它会把“这个 query 对所有 key 的分数”归一化成概率：

attn_scores[b, h, q, :] -> softmax

其中：

attn_scores[b, h, q, :].shape = (key_len,)
attn_probs[b, h, q, :].shape  = (key_len,)

并且这一行概率和为 1：

sum(attn_probs[b, h, q, :]) = 1

softmax 不改变 tensor 的整体形状，所以结果仍然是：

attn_probs.shape = (batch_size, num_heads, query_len, key_len)

在 self-attention 中通常 query_len == key_len == seq_length，所以就是：

(batch_size, num_heads, seq_length, seq_length)
直觉上：每个 token 作为 query 时，对整句话里所有 token 分配一组注意力权重。

补充问题六：contiguous() 是什么作用？

在 combine_heads 中：

def combine_heads(self, x):
    batch_size, _, seq_length, d_k = x.size()
    return x.transpose(1, 2).contiguous().view(batch_size, seq_length, self.d_model)

contiguous() 的作用是把一个内存不连续的 tensor 重新整理成内存连续的 tensor，这样后面的 view(...) 才能安全工作。

transpose(1, 2) 会把 shape 从：

(batch_size, num_heads, seq_length, d_k)

变成：

(batch_size, seq_length, num_heads, d_k)

但是 PyTorch 的 transpose 通常不会真的复制数据并重新排列内存，它只是改变 tensor 的视图和 stride。因此，看起来维度换了，但底层内存排列可能还是原来的顺序。

而 view(...) 要求 tensor 在内存中是连续排列的，否则它不知道怎么按照新 shape 正确解释内存。所以这里需要：

x.transpose(1, 2).contiguous().view(batch_size, seq_length, self.d_model)

把：

(batch_size, seq_length, num_heads, d_k)

合并成：

(batch_size, seq_length, d_model)

其中：

d_model = num_heads * d_k
transpose 改变维度视图，但可能导致内存不连续；contiguous 重新整理内存；view 依赖连续内存来 reshape。

现在很多时候也可以写成：

x.transpose(1, 2).reshape(batch_size, seq_length, self.d_model)

因为 reshape 在需要时会自动复制成连续内存。但教学代码里经常写成 transpose(...).contiguous().view(...)，这样更明确地表达：先换维度，再保证内存连续，再合并 head。

补充问题七：PyTorch 有没有 SwiGLU？如何用 SwiGLU 实现 FFN？

PyTorch 标准库里有 nn.GLU，也有 F.silu / nn.SiLU，但通常没有一个专门叫 nn.SwiGLU 的模块。SwiGLU 一般自己用两个 Linear 加 SiLU 实现。

经典 Transformer FFN 是：

FFN(x) = W2(activation(W1(x)))

SwiGLU FFN 是：

FFN(x) = W_down(SiLU(W_gate(x)) * W_up(x))

也就是先做两个并行投影：一个产生 gate，一个产生 value，然后用 SiLU(gate) * value 做逐元素门控。

import torch
from torch import nn
import torch.nn.functional as F

class SwiGLUFFN(nn.Module):
    def __init__(self, d_model, d_ff):
        super().__init__()
        self.w_gate = nn.Linear(d_model, d_ff)
        self.w_up = nn.Linear(d_model, d_ff)
        self.w_down = nn.Linear(d_ff, d_model)

    def forward(self, x):
        return self.w_down(F.silu(self.w_gate(x)) * self.w_up(x))

如果原来的 FFN 是：

class PositionWiseFeedForward(nn.Module):
    def __init__(self, d_model, d_ff):
        super().__init__()
        self.fc1 = nn.Linear(d_model, d_ff)
        self.fc2 = nn.Linear(d_ff, d_model)
        self.relu = nn.ReLU()

    def forward(self, x):
        return self.fc2(self.relu(self.fc1(x)))

可以替换成：

class PositionWiseSwiGLUFeedForward(nn.Module):
    def __init__(self, d_model, d_ff):
        super().__init__()
        self.gate_proj = nn.Linear(d_model, d_ff)
        self.up_proj = nn.Linear(d_model, d_ff)
        self.down_proj = nn.Linear(d_ff, d_model)

    def forward(self, x):
        x = F.silu(self.gate_proj(x)) * self.up_proj(x)
        return self.down_proj(x)

输入如果是：

(batch_size, seq_length, d_model)

那么中间 shape 是：

gate_proj(x): (batch_size, seq_length, d_ff)
up_proj(x):   (batch_size, seq_length, d_ff)
相乘后:       (batch_size, seq_length, d_ff)
down_proj:    (batch_size, seq_length, d_model)

注意参数量：SwiGLU 有三个 Linear，而普通 FFN 只有两个 Linear。所以很多 LLaMA 类模型会把 SwiGLU 的 hidden dim 调小，比如用 int(2/3 * 4 * d_model)，这样参数量和普通 4 * d_model FFN 接近。

补充问题八：PyTorch 的 SiLU 计算是什么？

torch.nn.SiLU / torch.nn.functional.silu 计算的是 Sigmoid Linear Unit，也叫 Swish。

公式是：

SiLU(x) = x * sigmoid(x)

其中：

sigmoid(x) = 1 / (1 + exp(-x))

所以：

SiLU(x) = x / (1 + exp(-x))

PyTorch 中：

import torch
import torch.nn.functional as F

y = F.silu(x)

等价于：

y = x * torch.sigmoid(x)

它和 ReLU 的区别是：ReLU 对负数直接截断成 0，而 SiLU 对负数是平滑压小，不是硬截断。

ReLU(x) = max(0, x)
SiLU(x) = x * sigmoid(x)

在 SwiGLU 里：

F.silu(gate_proj(x)) * up_proj(x)

意思是先用 SiLU 生成一个平滑的 gate，再用这个 gate 去逐元素调制另一条分支。

补充问题九：这是 Pre-LN 还是 Post-LN？RMSNorm 如何计算？

文章中的 DecoderLayer 代码是：

def forward(self, x, enc_output, src_mask, tgt_mask):
    attn_output = self.self_attn(x, x, x, tgt_mask)
    x = self.norm1(x + self.dropout(attn_output))
    attn_output = self.cross_attn(x, enc_output, enc_output, src_mask)
    x = self.norm2(x + self.dropout(attn_output))
    ff_output = self.feed_forward(x)
    x = self.norm3(x + self.dropout(ff_output))
    return x

这是 Post-LayerNorm（Post-LN），因为它的结构是：

x = LayerNorm(x + Sublayer(x))

也就是先做子层计算和残差连接，再做 LayerNorm。LayerNorm 放在残差连接之后，所以叫 Post-LN。

如果是 Pre-LayerNorm（Pre-LN），通常会写成：

norm_x = self.norm1(x)
attn_output = self.self_attn(norm_x, norm_x, norm_x, tgt_mask)
x = x + self.dropout(attn_output)

结构是：

x = x + Sublayer(LayerNorm(x))

也就是 LayerNorm 放在子层之前。

结构 公式 特点
Post-LN LayerNorm(x + Sublayer(x)) 原始 Transformer 常用；深层模型训练更容易不稳定。
Pre-LN x + Sublayer(Norm(x)) 现代深层 Transformer / LLM 更常用；梯度更容易沿残差路径传播。

现代主流 LLM 基本采用 Pre-LN，并且很多模型不用标准 LayerNorm，而使用 RMSNorm。LLaMA 风格大致是：

h = x + self_attn(rms_norm(x))
out = h + mlp(rms_norm(h))

RMSNorm 如何计算？

RMSNorm 的核心是：只用均方根 RMS 做缩放，不减均值。

给定输入向量：

x = [x1, x2, ..., xd]

RMS 是：

RMS(x) = sqrt((1/d) * sum(x_i^2))

RMSNorm 计算：

RMSNorm(x) = x / sqrt(mean(x^2) + eps) * weight

也就是：

y_i = x_i / sqrt((1/d) * sum_j(x_j^2) + eps) * g_i

其中 g_i 是可学习缩放参数 weight，eps 是防止除 0 的小常数。

PyTorch 实现大概是：

class RMSNorm(nn.Module):
    def __init__(self, dim, eps=1e-6):
        super().__init__()
        self.weight = nn.Parameter(torch.ones(dim))
        self.eps = eps

    def forward(self, x):
        rms = torch.sqrt(torch.mean(x * x, dim=-1, keepdim=True) + self.eps)
        x = x / rms
        return self.weight * x

也常写成更高效的形式：

x = x * torch.rsqrt(x.pow(2).mean(dim=-1, keepdim=True) + eps)

其中 rsqrt(z) 表示：

rsqrt(z) = 1 / sqrt(z)

LayerNorm 和 RMSNorm 的区别：

Norm 计算方式 是否减均值
LayerNorm (x - mean(x)) / sqrt(var(x) + eps) 是
RMSNorm x / sqrt(mean(x^2) + eps) 否
总结：文章代码是 Post-LN；现代主流 LLM 多用 Pre-LN，并常用 RMSNorm 替代 LayerNorm，以提升深层模型训练稳定性并减少一点计算。

1. PyTorch 广播规则：右对齐

PyTorch（NumPy 也是一样）的广播规则是从最右边的维度开始对齐，逐维比较，要求每一维要么相等，要么其中之一是 1。

以"A: [n] 与 B: [d, n, n, d]，且 d ≠ n"为例：

A:           [         n]
B: [d, n, n, d]   ← 最后一维是 d，与 A 的 n 不相等，且都不是 1

结果：直接报错。

RuntimeError: The size of tensor a (n) must match the size of tensor b (d)
at non-singleton dimension 3

如果想让 [n] 和 B 的某个中间维度 n 对齐，必须手动 reshape：

A.view(n, 1, 1) * B          # 对齐到倒数第 3 维的 n
A.view(1, n, 1, 1) * B       # 对齐到倒数第 4 维（前面用 1 补齐）
PyTorch 不会"智能匹配"同名维度。[n] 只能对齐到最后一维，其他位置必须显式 view 或 unsqueeze。这是写 attention、mask 广播时最常踩的坑之一。

2. torch.sqrt vs torch.rsqrt

函数计算含义
torch.sqrt(x)√x平方根
torch.rsqrt(x)1 / √xreciprocal sqrt，平方根的倒数

数学上 rsqrt(x) == 1 / sqrt(x)，但实际有几点区别：

  1. 性能：GPU/CPU 都有专门的 rsqrt 硬件指令（NVIDIA rsqrt.approx、x86 RSQRTSS），一条指令出结果；而 1/sqrt(x) 需要 sqrt + div 两步，div 又是 GPU 上较慢的操作。
  2. 数值精度：硬件 rsqrt 通常是"近似 + 一步 Newton 迭代"，精度可能略低，但深度学习里完全够用。
  3. 定义域：sqrt(0) = 0；rsqrt(0) = +inf。所以 RMSNorm 一定要加 eps。

RMSNorm 的两种写法：

# 写法 A：直观但慢
rms = torch.sqrt(torch.mean(x*x, dim=-1, keepdim=True) + eps)
x = x / rms

# 写法 B：快且数值稳定（推荐，工业标准）
x = x * torch.rsqrt(torch.mean(x*x, dim=-1, keepdim=True) + eps)
LLaMA、FlashAttention、PyTorch 官方 LayerNorm CUDA kernel 内部都用 rsqrt。归一化层里看到 x / sqrt(...)，几乎都应改成 x * rsqrt(...)。

3. Causal mask 的语义与 ~mask

生成 causal mask

seq_len = x.size(1)
mask = torch.tril(torch.ones(seq_len, seq_len, device=x.device, dtype=torch.bool))
mask = mask.view(1, 1, seq_len, seq_len)   # 广播到 [B, n_heads, S, S]
  • torch.tril 生成下三角矩阵：第 i 行只有列 j ≤ i 的位置为 1（True）。
  • 本约定下：True = 要保留的位置（合法可见的过去 token）。
  • 形状 [1, 1, S, S] 可以广播到 [B, n_heads, S, S]，与 attn_scores 对齐。
  • 用 dtype=torch.bool 比 float 更省内存，也避免 mask == 0 的浮点比较。

应用 mask 时为何要 ~mask？

masked_fill(condition, value) 的语义是：condition 为 True 的位置填 value。因此要"屏蔽未来 token"，要把 mask 反过来：

if mask is not None:
    attn_scores = attn_scores.masked_fill(~mask, float('-inf'))
  • ~ 是按位取反，对 bool tensor 来说就是 True/False 互换。
  • ~mask 之后，"上三角的未来 token 位置" 变成 True，被填成 -inf；softmax 后这些位置概率为 0。
最容易踩的坑：直接写 masked_fill(mask, -inf)，会把"要保留的位置"全部填成 -inf，整行变 -inf 触发 NaN，模型完全坏掉。一定要确认 mask 的语义约定。

两种约定都常见

约定True 含义典型来源SDPA 内部写法
ATrue = 保留PyTorch F.scaled_dot_product_attentionmasked_fill(~mask, -inf)
BTrue = 屏蔽HuggingFace 4D causal maskmasked_fill(mask, -inf)

两种都可以，关键是在 docstring 里写清楚约定，否则使用者很容易传反。

填 -inf 还是 -1e9？

之前的版本用 -1e9，在 fp16 下会溢出（fp16 范围约 ±65504）。推荐：

attn_scores.masked_fill(~mask, float('-inf'))
# 或者更通用：
attn_scores.masked_fill(~mask, torch.finfo(attn_scores.dtype).min)

4. Attention dropout vs Residual dropout

这两个 dropout 位置不同、作用不同，常常被混淆：

类型位置丢弃对象直觉
Attention dropout softmax 之后、与 V 相乘之前 注意力权重 P[i, j] 随机让"第 i 个 token 看不见第 j 个 token"，防止过度依赖单一 attention 边
Residual dropout 子层输出加到残差之前（x + dropout(sublayer(x))） 整层的输出向量 随机让"这一层完全不贡献"，防止过度依赖某一层

是否需要 attention dropout？

  • 原始 Transformer / BERT / GPT-2：两个都加。
  • 现代 LLaMA 风格大模型：通常两个都不加（大数据 + 大模型不需要 dropout，反而拖慢训练）。
  • 小数据微调 / 学术实验：可以两个都加，正则化更强。

如果要加，代码改动很小：

# __init__ 里
self.attn_dropout = nn.Dropout(dropout)

# scaled_dot_product_attention 里，softmax 之后
attn_probs = torch.softmax(attn_scores, dim=-1)
attn_probs = self.attn_dropout(attn_probs)   # ← 新增
output = torch.matmul(attn_probs, v)

5. KV Cache：为什么朴素 decode 会浪费

问题

自回归生成时，假设已生成 [t1, t2, t3]，现在要生成 t4：

输入 [t1, t2, t3]
→ K = W_k · [t1, t2, t3]     # 3 个 key
→ V = W_v · [t1, t2, t3]     # 3 个 value
→ out = softmax(QK^T)V       # 输出 3 个 token 的表示
→ 只取最后一个 token 过 LM head，采样得到 t4

下一步生成 t5 时：

输入 [t1, t2, t3, t4]
→ K = W_k · [t1, t2, t3, t4]   # t1,t2,t3 的 K 又算了一遍！
→ V = W_v · [t1, t2, t3, t4]   # 同样浪费

朴素方式生成 N 个 token 的总计算量是 O(N³)，其中大量重复计算。

KV Cache 的核心观察

  1. 由于 causal mask，token i 的输出只依赖 j ≤ i 的 K、V。
  2. 过去 token 的 K、V 一旦算出永远不变，缓存即可。
  3. 过去 token 的 Q 是没用的（它们的输出我们已经用过、采样过了）。
  4. 生成新 token 时，只需：算新 token 的 Q、K、V → 把新 K、V 拼接到缓存 → 新 Q 对完整缓存 K、V 做 attention。
朴素KV Cache
每步计算量O(N²)O(N)
生成 N 个 token 总量O(N³)O(N²)
显存O(N)O(N)（需多存 K、V）

代码改造

def forward(self, x, kv_cache=None):
    # 训练时 x 是整个序列；推理时 x 只有 1 个新 token
    q = self.split_heads(self.q_linear(x))   # [B, H, S_new, D]
    k = self.split_heads(self.k_linear(x))
    v = self.split_heads(self.v_linear(v))

    if kv_cache is not None:
        past_k, past_v = kv_cache
        k = torch.cat([past_k, k], dim=2)    # 沿 seq 维拼接
        v = torch.cat([past_v, v], dim=2)
    new_cache = (k, v)

    # 推理 step 时 Q 长度=1，KV 长度=N，不需要 causal mask
    attn = self.sdpa(q, k, v, mask=None)
    return self.output_linear(self.combine_heads(attn)), new_cache

工业级延伸

  • PagedAttention (vLLM)：像操作系统的虚拟内存一样管理 KV cache，解决长度不一的 batch 浪费。
  • Sliding Window (Mistral)：只缓存最近 N 个 token，固定显存。
  • GQA / MQA (LLaMA-2 70B / Falcon)：多个 Q head 共享 K、V，cache 大幅缩小。
  • 量化 KV cache：把 cache 存成 int8 / fp8，省一半显存。
  • FlashAttention (2205.14135)：训练和推理 prefill 阶段的标准实现，避免落盘 N×N 矩阵。

6. SwiGLU FFN 的 d_ff 应该取多少

普通 FFN（2 个 Linear）的惯例：

d_ff = 4 * d_model

SwiGLU FFN 因为多了 w_gate，相同 d_ff 下参数量是普通 FFN 的 1.5 倍（3 个 Linear vs 2 个）。为对齐参数量，LLaMA 把 d_ff 缩成：

d_ff = (2/3) * 4 * d_model = (8/3) * d_model ≈ 2.67 * d_model

再向上取整到 GPU 友好的倍数（通常 256）：

def swiglu_ff_dim(d_model, multiple_of=256):
    d_ff = int(8 * d_model / 3)
    d_ff = ((d_ff + multiple_of - 1) // multiple_of) * multiple_of
    return d_ff

真实模型的取值参考

模型d_modeld_ff比例
LLaMA-1 7B4096110082.69×
LLaMA-2 7B4096110082.69×
LLaMA-2 70B8192286723.5×
LLaMA-3 8B4096143363.5×
Mistral 7B4096143363.5×
Qwen2 7B3584189445.29×

趋势是比 2.67× 更大（3.5× 甚至 5×），稍宽的 FFN 收益更好。

实战建议：
  • 保守 / 对齐普通 FFN 参数量：d_ff = round_to(8/3 * d_model, 256)
  • 跟随现代 LLM：直接 d_ff = 3.5 * d_model 再 round 到 256 的倍数
  • 练手：d_ff = 4 * d_model 也完全 OK，只是参数多一点点

7. 最终可训练的 decoder-only layer（迭代结果）

经过若干轮 review 与修复（-1e9 → -inf、加 self.eps、if not mask → if mask is None、修正 mask 取反语义等），最终版本如下：

import torch as T
from torch import nn
import torch.nn.functional as F


class MultiHeadAttention(nn.Module):
  def __init__(self, d_model, n_heads):
    super().__init__()
    self.d_model = d_model
    self.n_heads = n_heads
    self.head_dim = d_model // n_heads
    assert self.head_dim * n_heads == d_model
    self.q_linear = nn.Linear(d_model, d_model)
    self.k_linear = nn.Linear(d_model, d_model)
    self.v_linear = nn.Linear(d_model, d_model)
    self.output_linear = nn.Linear(d_model, d_model)

  def split_heads(self, x):
    batch_size, seq_len, d_model = x.size()
    return x.view(batch_size, seq_len, self.n_heads, self.head_dim).transpose(1, 2)

  def scaled_dot_product_attention(self, q, k, v, mask=None):
    """mask: bool tensor broadcastable to [B, n_heads, S, S].
       True = keep, False = mask out (filled with -inf)."""
    attn_scores = T.matmul(q, k.transpose(-2, -1)) / (self.head_dim ** 0.5)
    if mask is not None:
      attn_scores = attn_scores.masked_fill(~mask, float('-inf'))
    attn_probs = T.softmax(attn_scores, dim=-1)
    return T.matmul(attn_probs, v)

  def combine_heads(self, x):
    batch_size, _, seq_len, head_dim = x.size()
    return x.transpose(1, 2).contiguous().view(batch_size, seq_len, self.d_model)

  def forward(self, q, k, v, mask=None):
    q = self.split_heads(self.q_linear(q))
    k = self.split_heads(self.k_linear(k))
    v = self.split_heads(self.v_linear(v))
    attn_output = self.scaled_dot_product_attention(q, k, v, mask)
    return self.output_linear(self.combine_heads(attn_output))


class SwiGLUFFN(nn.Module):
  def __init__(self, d_model, d_ff):
    super().__init__()
    self.w_up = nn.Linear(d_model, d_ff)
    self.w_down = nn.Linear(d_ff, d_model)
    self.w_gate = nn.Linear(d_model, d_ff)

  def forward(self, x):
    return self.w_down(F.silu(self.w_up(x)) * self.w_gate(x))


class RMSNorm(nn.Module):
  def __init__(self, dim, eps=1e-6):
    super().__init__()
    self.weight = nn.Parameter(T.ones(dim))
    self.eps = eps

  def forward(self, x):
    x = x * T.rsqrt(T.mean(x * x, dim=-1, keepdim=True) + self.eps)
    return self.weight * x


class TransformerDecoderOnlyLayer(nn.Module):
  def __init__(self, d_model, n_heads, d_ff, dropout=0.1):
    super().__init__()
    self.self_attn = MultiHeadAttention(d_model, n_heads)
    self.ffn = SwiGLUFFN(d_model, d_ff)
    self.norm1 = RMSNorm(d_model)
    self.norm2 = RMSNorm(d_model)
    self.dropout1 = nn.Dropout(dropout)
    self.dropout2 = nn.Dropout(dropout)

  def forward(self, x, mask=None):
    if mask is None:
      seq_len = x.size(1)
      mask = T.tril(T.ones(seq_len, seq_len, device=x.device, dtype=T.bool))
      mask = mask.view(1, 1, seq_len, seq_len)
    norm_x = self.norm1(x)
    x = x + self.dropout1(self.self_attn(norm_x, norm_x, norm_x, mask))
    norm_x = self.norm2(x)
    x = x + self.dropout2(self.ffn(norm_x))
    return x

迭代过程中踩过的坑总结

问题修复
RMSNorm 没用 eps，全 0 输入会 NaN+ self.eps
RMSNorm 用 sqrt 再除，慢且容易写错用 x * rsqrt(...)
mask 填 -1e9，fp16 溢出填 float('-inf') 或 finfo(dtype).min
外面调用者必须自备 causal mask，容易忘layer 内部 if mask is None 自动生成 tril
if not mask: 对 tensor 报 "Boolean value ambiguous"改成 if mask is None:
mask 语义反了：tril 给的是"保留位置"，但 masked_fill(mask, -inf) 把保留位置填掉masked_fill(~mask, -inf) 或改用 triu 生成"屏蔽位置"
缺少 head_dim * n_heads == d_model 校验加 assert

还差什么才是完整可上线的 decoder layer

  • 位置编码 / RoPE：当前实现对 token 顺序完全不敏感，必须在 embedding 层或 attention 内部加入。
  • KV Cache 支持：推理必备（见 KV Cache 节）。
  • 多层堆叠 + Embedding + LM head：组成完整 model 的外层包装。
  • 用 F.scaled_dot_product_attention 替换手写 attention：PyTorch 2.0+ 自动使用 FlashAttention，更快更省显存。

8. RotaryPositionEmbedding (RoPE) 实现

核心思想

RoPE 不是"加"位置编码（不像 sinusoidal 那样 x + PE），而是对 Q 和 K 做旋转，把位置 m 编码成旋转角度。

关键性质：旋转后的 <q_m, k_n> 内积只取决于相对位置 m - n，所以模型天然学到的是相对位置关系，而不是绝对位置。数学上：

<R_m · q,  R_n · k>  =  q^T · R_m^T · R_n · k  =  q^T · R_{n-m} · k

（旋转矩阵满足 R_m^T · R_n = R_{n-m}）

数学公式

把每个 head 的 d 维向量两两分组成 d/2 个二维向量。对位置 m 的第 i 组（i = 0, 1, ..., d/2-1），用角度 θ_i = 10000^(-2i/d)，旋转角度为 m · θ_i：

[x'_{2i}  ]   [cos(mθ_i)  -sin(mθ_i)] [x_{2i}  ]
[x'_{2i+1}] = [sin(mθ_i)   cos(mθ_i)] [x_{2i+1}]

越靠后的维度（i 大）频率越低（θ_i 越小），形成多尺度的位置编码。

两种 layout

方式配对典型来源
A：相邻配对(x0, x1), (x2, x3), ...RoFormer 原论文
B：前后半配对(x_i, x_{i+d/2})LLaMA 实际实现，PyTorch 友好

两种数学等价，下面用 LLaMA 风格（方式 B）。

实现代码

class RotaryPositionEmbedding(nn.Module):
    """
    head_dim 必须是偶数。
    用法：
        rope = RotaryPositionEmbedding(head_dim, max_seq_len=8192)
        q, k = rope(q, k)   # q, k shape: [B, n_heads, S, head_dim]
    """
    def __init__(self, head_dim, max_seq_len=8192, base=10000.0):
        super().__init__()
        assert head_dim % 2 == 0
        self.head_dim = head_dim

        # 频率：θ_i = base^(-2i/d)
        inv_freq = 1.0 / (base ** (T.arange(0, head_dim, 2).float() / head_dim))

        # 预计算所有位置的 cos/sin
        positions = T.arange(max_seq_len).float()
        freqs = T.outer(positions, inv_freq)            # [max_seq_len, d/2]
        emb = T.cat([freqs, freqs], dim=-1)             # [max_seq_len, d]
        # 方式 B：前后半用同一组角度

        self.register_buffer("cos_cached", emb.cos(), persistent=False)
        self.register_buffer("sin_cached", emb.sin(), persistent=False)

    @staticmethod
    def _rotate_half(x):
        """把后半段取负放到前面：[a, b] → [-b, a]"""
        x1 = x[..., : x.shape[-1] // 2]
        x2 = x[..., x.shape[-1] // 2 :]
        return T.cat([-x2, x1], dim=-1)

    def forward(self, q, k, offset=0):
        """
        q, k: [B, n_heads, S, head_dim]
        offset: 推理用 KV cache 时，新 token 的位置 = past_len
        """
        seq_len = q.size(-2)
        cos = self.cos_cached[offset : offset + seq_len].unsqueeze(0).unsqueeze(0)
        sin = self.sin_cached[offset : offset + seq_len].unsqueeze(0).unsqueeze(0)
        q_rot = q * cos + self._rotate_half(q) * sin
        k_rot = k * cos + self._rotate_half(k) * sin
        return q_rot, k_rot

集成到 MultiHeadAttention

class MultiHeadAttention(nn.Module):
    def __init__(self, d_model, n_heads, max_seq_len=8192):
        super().__init__()
        ...
        self.rope = RotaryPositionEmbedding(self.head_dim, max_seq_len)

    def forward(self, q, k, v, mask=None, pos_offset=0):
        q = self.split_heads(self.q_linear(q))
        k = self.split_heads(self.k_linear(k))
        v = self.split_heads(self.v_linear(v))

        # ← 在 split_heads 之后、attention 之前应用 RoPE
        # 注意：只对 Q 和 K 做，V 不需要
        q, k = self.rope(q, k, offset=pos_offset)

        attn = self.scaled_dot_product_attention(q, k, v, mask)
        return self.output_linear(self.combine_heads(attn))

关键细节

  1. 为什么 _rotate_half 是 [-x2, x1]？二维旋转矩阵展开后，前半 = a·cos − b·sin，后半 = a·sin + b·cos。写成向量形式就是 q·cos + [−b, a]·sin。
  2. 应用时机：必须在 split_heads 之后、scaled dot product 之前，因为旋转是按 head_dim 内的成对维度做。
  3. KV cache 的 offset：推理时新 token 的位置 = 缓存里已有 token 数；否则会被当成位置 0，角度全错。
  4. dtype：cos/sin 用 fp32 算更精确，可在旋转前 .float()，旋转后 .to(q.dtype)。

方法做法
Position Interpolation (PI)把位置 m 缩放成 m · L_train / L_target，等于压缩角度
NTK-aware scaling调大 base，让高频部分基本不变，低频被拉伸

9. RoPE 应在 linear 之前还是之后？只 apply 到 Q、K 吗？

结论：

  • RoPE 应在 q_linear / k_linear 之后、且在 split_heads 之后、attention 之前 应用。
  • 只对 Q 和 K 做，不对 V 做（但 v_linear 还是必须有）。

处理顺序（每层 attention 内部）

x → q_linear / k_linear / v_linear
  → split_heads
  → RoPE（仅 Q、K）   ← 在这里
  → scaled dot product attention
  → combine_heads
  → output_linear

为什么必须在 linear 之后？

RoPE 旋转的对象是 Q、K 这两个 attention 中真正参与内积的向量。q_linear / k_linear 不只是改维度，更是学习"从 x 中提取什么作为 query / key"。RoPE 的相对位置魔法依赖：

<q_m, k_n> ≈ 相对位置 (n - m) 的函数

这里的 q_m、k_n 必须是 q_linear(x_m) 和 k_linear(x_n) 之后的向量。如果在 linear 之前旋转 x，那线性变换会把旋转结构彻底打乱（旋转矩阵与任意矩阵不可交换），RoPE 就失效了。

为什么必须在 split_heads 之后？

RoPE 是按 head_dim 内的成对维度做 2D 旋转。必须"看到 head_dim"才能正确分组：每个 head 独立做旋转，不同 head 之间互不影响。如果在 split_heads 之前做，旋转分组会跨越 head 边界，破坏 multi-head 的独立性。

为什么不旋转 V（但仍需 v_linear）

见 为什么 V 不旋转：V 在 attention 中是被加权求和，没有内积参与"相对位置抵消"的数学魔法。但 v_linear(x) 必须做 —— 它学习"从 x 中提取什么作为 value（被加权求和的内容）"。Q、K、V 三个投影是独立学习的，分别对应"用什么 query / 被 query / 输出什么内容"。

各组件应不应该 apply 总览

linearsplit_headsRoPEattention
Q✅✅✅✅
K✅✅✅✅
V✅✅❌✅
RoPE 是 attention 内部的"位置插件"：必须作用在已经投影、已经分头的 Q、K 上，V 完全不动。它与 q_linear/k_linear/v_linear 是协作关系，不是替代关系。

10. 为什么 RoPE 每层都要做（与原始加法式 PE 的区别）

原始 Transformer 的 sinusoidal PE 只在输入层加一次（x = embedding + PE），位置信号跟着 residual 流过每一层。但 RoPE 必须每层 attention 都重新做，原因如下：

加法式 PERoPE
应用对象hidden stateQ、K（V 不变）
应用位置embedding 之后每层 attention 内部
信号传播通过 residual 一直带下去不存到 hidden state 里
每层都要重做？否是

核心原因：RoPE 不修改 hidden state，它只修改 Q 和 K。Attention 计算完后，output_linear(attn(Q_rot, K_rot, V)) 的结果已经不带旋转 —— 因为 attention 算完后 Q、K 不再出现，只剩下 V 的加权和。

  • 下一层拿到的 hidden state 里，没有任何"我是位置 m"的信号。
  • 所以下一层算 attention 时，必须重新给 Q、K 应用 RoPE。

为什么不能像加法 PE 那样"加一次就够"？

  1. hidden state 是 V 的加权和，已经混合了不同位置的 token 信息，再去"旋转"它没有清晰的位置语义。
  2. 后面的 Linear（FFN、output_linear）会破坏旋转：旋转矩阵和任意矩阵不可交换，过一个 FFN 后旋转角度就乱了。

而加法式 PE 是"x + PE"，PE 像一个信号附加在向量上，经过线性变换后还能部分保留（虽然也会被稀释）。

加法式 PE 把位置信息"写入" hidden state 跟着 residual 传；RoPE 只在 attention 算分时"临时"给 Q、K 添加位置感，不进入 hidden state，所以每层必须重做。这是设计哲学的差异，不是冗余，计算代价也很小（只是一次 element-wise 乘加）。

11. KV cache 缓存的是 RoPE 之前还是之后的 K？

缓存的是 RoPE 之后的 K（V 没有 RoPE，原样缓存）。

原因

每个 token 在序列中的位置是固定的：位置 5 的 token 它的 K 旋转角度永远是 5 · θ_i。所以 RoPE 旋转后的 K 一旦算出来就不会变，缓存"已旋转的 K"完全合法。

反过来缓存"旋转前的 K"，每次生成新 token 都要把所有历史 K 重新旋转一遍 —— 浪费计算，违背 KV cache 初衷。

正确流程（生成第 N+1 个 token）

# 1. 新 token 过 linear 得到原始 Q、K、V（seq_len=1）
q_new = q_linear(x_new)
k_new = k_linear(x_new)
v_new = v_linear(x_new)

# split_heads → [B, n_heads, 1, head_dim]

# 2. 对新 Q、新 K 应用 RoPE，offset = N
q_new, k_new = rope(q_new, k_new, offset=N)

# 3. 拼接到缓存（缓存里的 K 已经是旋转过的！）
k_full = torch.cat([past_k_rotated, k_new], dim=2)  # [B, H, N+1, D]
v_full = torch.cat([past_v, v_new], dim=2)

# 4. attention：新 Q（已旋转，位置 N）vs 所有历史 K（已旋转，位置 0..N）
out = attention(q_new, k_full, v_full)

# 5. 更新缓存
new_cache = (k_full, v_full)
一句话：位置固定 → 旋转结果固定 → 缓存旋转后的 K 一劳永逸。V 不需要 RoPE，原样缓存。Q 不缓存。

12. 为什么 V 不一起旋转

常见疑问：既然 Q、K 都旋转了，把 V 也旋转，attention 输出不就"带旋转"了吗？

答案：不能旋转 V，因为 V 在 attention 中没有内积参与"相对位置抵消"的数学魔法，旋转它只会把绝对位置的角度噪声写进 hidden state，反而污染下一层。

RoPE 工作的关键：内积保相对位置

RoPE 之所以工作，是因为旋转矩阵满足：

<R_m · q,  R_n · k>  =  q^T · R_{n-m} · k

也就是说 旋转后的 Q、K 做内积，自动得到只依赖相对位置 (n-m) 的 attention 分数。这是 RoPE 全部的"位置信号注入"机制。

V 的角色完全不同

attention 的最后一步：

output_i = Σ_j  attn_weights[i, j] · v_j

V 是被加权求和的内容，没有任何"内积 → 位置相对性"环节。

如果你把 V 也旋转：

output_i = Σ_j  attn_weights[i, j] · (R_j · v_j)
        = 一堆不同角度旋转过的 v 向量的加权和

结果是一个混合了多个绝对位置旋转角度的向量。这个向量：

  • 不再具有 RoPE 的相对位置性质（内积里抵消的魔法不适用于求和）
  • 把"绝对位置 j 的旋转"残留在了 hidden state 里
  • 下一层拿到这个 hidden state 后再过 Linear，那些位置残留会被打乱
  • 然后下一层又要再次 RoPE，两次旋转叠加，位置语义彻底崩坏

加法式 PE 能"持续传"，为什么旋转不行？

  • 加法 x + PE 是叠加：经过线性变换后 PE 部分还能在向量里保留近似信号。
  • 旋转 R · x 是改变方向：经过任意 Linear 后旋转角度会被矩阵打乱。所以旋转不能"传"，只能"用一次就丢"。
RoPE 通过"Q、K 旋转后内积 = 相对位置函数"工作；V 没有内积参与这个魔法，旋转它只会破坏下一层。不旋转 V 恰好保证 hidden state 的"位置中立"，每层可以从零开始用 RoPE 重新注入位置。

总结

split_heads 里的 transpose(1, 2) 是为了把张量从：

(batch_size, seq_length, num_heads, d_k)

转换为：

(batch_size, num_heads, seq_length, d_k)

这样每个 attention head 都可以独立地对完整序列计算 self-attention。它是 Multi-Head Attention 能正确并行计算多个 head 的关键步骤。
